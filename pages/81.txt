<p>In <a href="/wiki/Computer_programming" title="Computer programming">computer programming</a>, <b>data-driven programming</b> is a <a href="/wiki/Programming_paradigm" title="Programming paradigm">programming paradigm</a> in which the program statements describe the data to be matched and the processing required rather than defining a sequence of steps to be taken.<sup id="cite&#95;ref-awk&#95;1-0" class="reference"><a href="#cite_note-awk-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> Standard examples of data-driven languages are the text-processing languages <a href="/wiki/Sed" title="Sed">sed</a> and <a href="/wiki/AWK" title="AWK">AWK</a>,<sup id="cite&#95;ref-awk&#95;1-1" class="reference"><a href="#cite_note-awk-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> and the document transformation language <a href="/wiki/XSLT" title="XSLT">XSLT</a>, where the data is a sequence of lines in an <a href="/wiki/Input_stream" class="mw-redirect" title="Input stream">input stream</a> – these are thus also known as line-oriented languages – and pattern matching is primarily done via <a href="/wiki/Regular_expression" title="Regular expression">regular expressions</a> or line numbers.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Related_paradigms">Related paradigms</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data-driven_programming&amp;action=edit&amp;section=1" title="Edit section: Related paradigms"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Data-driven programming is similar to <a href="/wiki/Event-driven_programming" title="Event-driven programming">event-driven programming</a>, in that both are structured as pattern matching and resulting processing, and are usually implemented by a <a href="/wiki/Main_loop" class="mw-redirect" title="Main loop">main loop</a>, though they are typically applied to different domains. The condition/action model is also similar to <a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">aspect-oriented programming</a>, where when a <a href="/wiki/Join_point" title="Join point">join point</a> (condition) is reached, a <a href="/wiki/Pointcut" title="Pointcut">pointcut</a> (action) is executed. A similar paradigm is used in some <a href="/wiki/Tracing_(software)" title="Tracing (software)">tracing</a> frameworks such as <a href="/wiki/DTrace" title="DTrace">DTrace</a>, where one lists probes (instrumentation points) and associated actions, which execute when the condition is satisfied.
</p><p>Adapting <a href="/wiki/Abstract_data_type" title="Abstract data type">abstract data type</a> design methods to <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> results in a data-driven design.<sup id="cite&#95;ref-responsibility&#95;2-0" class="reference"><a href="#cite_note-responsibility-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> This type of design is sometimes used in object-oriented programming to define classes during the conception of a piece of software.
</p>
<div class="mw-heading mw-heading2"><h2 id="Applications">Applications</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data-driven_programming&amp;action=edit&amp;section=2" title="Edit section: Applications"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Data-driven programming is typically applied to streams of structured data, for filtering, transforming, aggregating (such as computing statistics), or calling other programs. Typical streams include <a href="/wiki/Computer_data_logging" class="mw-redirect" title="Computer data logging">log files</a>, <a href="/wiki/Delimiter-separated_values" title="Delimiter-separated values">delimiter-separated values</a>, or email messages, notably for <a href="/wiki/Email_filtering" title="Email filtering">email filtering</a>. For example, an AWK program may take as input a stream of log statements, and for example send all to the console, write ones starting with WARNING to a "WARNING" file, and send an email to a <a href="/wiki/Sysadmin" class="mw-redirect" title="Sysadmin">sysadmin</a> in case any line starts with "ERROR". It could also record how many warnings are logged per day. Alternatively, one can process streams of delimiter-separated values, processing each line or aggregated lines, such as the sum or max. In email, a language like <a href="/wiki/Procmail" title="Procmail">procmail</a> can specify conditions to match on some emails, and what actions to take (deliver, bounce, discard, forward, etc.).
</p><p>Some data-driven languages are <a href="/wiki/Turing-complete" class="mw-redirect" title="Turing-complete">Turing-complete</a>, such as AWK and even sed, while others are intentionally very limited, notably for filtering. An extreme example of the latter is <a href="/wiki/Pcap" title="Pcap">pcap</a>, which only consists of filtering, with the only action being "capture". Less extremely, <a href="/wiki/Sieve_(mail_filtering_language)" title="Sieve (mail filtering language)">sieve</a> has filters and actions, but in the base standard has no variables or loops, only allowing stateless filtering statements: each input element is processed independently. Variables allow state, which allow operations that depend on more than one input element, such as aggregation (summing inputs) or <a href="/wiki/Throttling_process_(computing)" class="mw-redirect" title="Throttling process (computing)">throttling</a> (allow at most 5 mails per hour from each sender, or limiting repeated log messages).
</p><p>Data-driven languages frequently have a default action: if no condition matches, line-oriented languages may print the line (as in sed), or deliver a message (as in sieve). In some applications, such as filtering, matching is may be done <i>exclusively</i> (so only <i>first</i> matching statement), while in other cases <i>all</i> matching statements are applied. In either case, failure to match <i>any</i> pattern may be "default behavior" or can be seen as an error, to be caught by a catch-all statement at the end.
</p>
<div class="mw-heading mw-heading2"><h2 id="Benefits_and_issues">Benefits and issues</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data-driven_programming&amp;action=edit&amp;section=3" title="Edit section: Benefits and issues"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>While the benefits and issues may vary between implementation, there are a few big potential benefits of and problems with this paradigm.  Functionality simply requires that it knows the <a href="/wiki/Abstract_data_type" title="Abstract data type">abstract data type</a> of the variables it is working with.  Functions and <a href="/wiki/Programming_interface" class="mw-redirect" title="Programming interface">interfaces</a> can be used on all objects with the same data fields, for instance the object's "position". Data can be grouped into objects or "entities" according to preference with little to no consequence.
</p><p>While data-driven design does prevent coupling of data and functionality, in some cases, data-driven programming has been argued to lead to bad <a href="/wiki/Object-oriented_design" class="mw-redirect" title="Object-oriented design">object-oriented design</a>, especially when dealing with more abstract data. This is because a purely data-driven object or entity is defined by the way it is <a href="/wiki/Data_representation" class="mw-redirect" title="Data representation">represented</a>.  Any attempt to change the structure of the object would immediately break the functions that rely on it.
</p><p>As an example, one might represent <a href="/wiki/Automotive_navigation_system" title="Automotive navigation system">driving directions</a> as a series of intersections (two intersecting streets) where the driver must turn right or left.  If an intersection (in the United States) is represented in data by the <a href="/wiki/Zip_code" class="mw-redirect" title="Zip code">zip code</a> (5-digit number) and two <a href="/wiki/Street_name" title="Street name">street names</a> (strings of text), bugs may appear when a city where streets <a href="/wiki/Intersection_(road)" title="Intersection (road)">intersect</a> multiple times is encountered.  While this example may be oversimplified, restructuring of data is a fairly common problem in software engineering, either to eliminate bugs, increase efficiency, or support new features.
</p>
