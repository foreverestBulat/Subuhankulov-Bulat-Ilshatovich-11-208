					<div id="contentSub"><div id="mw-content-subtitle"><span class="mw-redirectedfrom">(Redirected from <a href="/w/index.php?title=Web_application_security&amp;redirect=no" class="mw-redirect" title="Web application security">Web application security</a>)</span></div></div>
					
					
					<div id="mw-content-text" class="mw-body-content"><div class="mw-subjectpageheader">
</div><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Measures taken to improve the security of an application</div>
<p><b>Application security</b> (<b>AppSec</b>) includes all tasks that introduce a secure <a href="/wiki/Software_development_life_cycle" class="mw-redirect" title="Software development life cycle">software development life cycle</a> to development teams. Its final goal is to improve security practices and, through that, to find, fix and preferably prevent security issues within applications. It encompasses the whole application life cycle from requirements analysis, design, implementation, verification as well as maintenance.<sup id="cite&#95;ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p><p><b>Web application security</b> is a branch of <a href="/wiki/Information_security" title="Information security">information security</a> that deals specifically with the security of <a href="/wiki/Website" title="Website">websites</a>, <a href="/wiki/Web_application" title="Web application">web applications</a>, and <a href="/wiki/Web_service" title="Web service">web services</a>. At a high level, web application security draws on the principles of application security but applies them specifically to the <a href="/wiki/Internet" title="Internet">internet</a> and <a href="/wiki/World_Wide_Web" title="World Wide Web">web</a> systems.<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> The application security also concentrates on <a href="/wiki/Mobile_app" title="Mobile app">mobile apps</a> and their security which includes iOS and Android Applications
</p><p>Web Application Security Tools are specialized tools for working with HTTP traffic, e.g., <a href="/wiki/Web_application_firewall" title="Web application firewall">Web application firewalls</a>.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Approaches">Approaches</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Application_security&amp;action=edit&amp;section=1" title="Edit section: Approaches"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Different approaches will find different subsets of the security <a href="/wiki/Vulnerability" title="Vulnerability">vulnerabilities</a> lurking in an application and are most effective at different times in the software lifecycle. They each represent different tradeoffs of time, effort, cost and vulnerabilities found.
</p>
<ul><li><a href="/wiki/Design_review" title="Design review">Design review</a>. Before code is written the application's architecture and design can be reviewed for security problems. A common technique in this phase is the creation of a <a href="/wiki/Threat_model" title="Threat model">threat model</a>.</li>
<li><a href="/wiki/White-box_testing" title="White-box testing">White-box testing</a>, or <a href="/wiki/Code_review" title="Code review">code review</a>. Critical examination of internal structure, architecture, design, etc.</li>
<li><a href="/wiki/Black-box_testing" title="Black-box testing">Black-box testing</a>. Tests functionality rather than internal structure.</li>
<li>Automated Tooling. Many security tools can be automated through inclusion into the development or testing environment. Examples of those are automated DAST/SAST tools that are integrated into code editor or CI/CD platforms.</li>
<li><a href="/wiki/Bug_bounty_program" title="Bug bounty program">Coordinated vulnerability platforms</a>. These are hacker-powered application security solutions offered by many websites and software developers by which individuals can receive recognition and compensation for reporting bugs.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Security_threats">Security threats</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Application_security&amp;action=edit&amp;section=2" title="Edit section: Security threats"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The Open Worldwide Application Security Project (<a href="/wiki/OWASP" title="OWASP">OWASP</a>) provides free and open resources. It is led by a non-profit called The OWASP Foundation. The OWASP Top 10 - 2017 results from recent research based on comprehensive data compiled from over 40 partner organizations. This data revealed approximately 2.3 million vulnerabilities across over 50,000 applications.<sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> According to the OWASP Top 10 - 2021, the ten most critical web application security risks include:<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ol><li>Broken <a href="/wiki/Access_control#Computer_security" title="Access control">access control</a></li>
<li>Cryptographic failures</li>
<li><a href="/wiki/Code_injection" title="Code injection">Injection</a></li>
<li>Insecure design</li>
<li>Security misconfiguration</li>
<li>Vulnerable and outdated components</li>
<li>Identification and authentification failures</li>
<li>Software and data integrity failures</li>
<li>Security logging and monitoring failures*</li>
<li>Server-side request forgery (SSRF)*</li></ol>
<div class="mw-heading mw-heading2"><h2 id="Security_controls">Security controls</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Application_security&amp;action=edit&amp;section=3" title="Edit section: Security controls"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The <a rel="nofollow" class="external text" href="https://top10proactive.owasp.org/">OWASP Top 10 Proactive Controls 2024</a> is a list of security techniques every software architect and developer should know and heed.
</p><p>The current list contains:
</p>
<ol><li>Implement access control</li>
<li>Use cryptography the proper way</li>
<li>Validate all input &amp; handle exceptions</li>
<li>Address security from the start</li>
<li>Secure by default configurations</li>
<li>Keep your components secure</li>
<li>Implement digital identity</li>
<li>Use browser security features</li>
<li>Implement security logging and monitoring</li>
<li>Stop server-side request forgery</li></ol>
<div class="mw-heading mw-heading2"><h2 id="Tooling_for_security_testing">Tooling for security testing</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Application_security&amp;action=edit&amp;section=4" title="Edit section: Tooling for security testing"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Security testing techniques scour for vulnerabilities or security holes in applications. These vulnerabilities leave applications open to <a href="/wiki/Exploit_(computer_security)" title="Exploit (computer security)">exploitation</a>. Ideally, security testing is implemented throughout the entire <a href="/wiki/Software_development_life_cycle" class="mw-redirect" title="Software development life cycle">software development life cycle</a> (SDLC) so that vulnerabilities may be addressed in a timely and thorough manner.
</p><p>There are many kinds of automated tools for identifying vulnerabilities in applications. Common tool categories used for identifying application vulnerabilities include:
</p>
<ul><li><a href="/wiki/Static_application_security_testing" title="Static application security testing">Static application security testing</a> (SAST) analyzes source code for security vulnerabilities during an application's development. Compared to DAST, SAST can be utilized even before the application is in an executable state. As SAST has access to the full source code it is a white-box approach. This can yield more detailed results but can result in many false positives that need to be manually verified.</li>
<li><a href="/wiki/Dynamic_application_security_testing" title="Dynamic application security testing">Dynamic application security testing</a> (DAST, often called <a href="/wiki/Vulnerability_scanner" title="Vulnerability scanner">vulnerability scanners</a>) automatically detects vulnerabilities by crawling and analyzing websites. This method is highly scalable, easily integrated and quick. DAST tools are well suited for dealing with low-level attacks such as injection flaws but are not well suited to detect high-level flaws, e.g., logic or business logic flaws.<sup id="cite&#95;ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Fuzzing" title="Fuzzing">Fuzzing</a> tools are commonly used for input testing.<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup></li>
<li>Industry application security research highlights increasing risks related to insecure APIs, client-side code tampering, and runtime exploitation, reinforcing the importance of comprehensive dynamic and runtime security testing.<sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup></li>
<li>Interactive application security testing (IAST) assesses applications from within using software instrumentation. This combines the strengths of both SAST and DAST methods as well as providing access to code, HTTP traffic, library information, backend connections and configuration information.<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-OWASP&#95;IAST&#95;11-0" class="reference"><a href="#cite_note-OWASP_IAST-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> Some IAST products require the application to be attacked, while others can be used during normal quality assurance testing.<sup id="cite&#95;ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:RS#Questionable_sources" class="mw-redirect" title="Wikipedia:RS"><span title="The material in the vicinity of this tag may rely on a promotional source. (December 2018)">promotional source?</span></a></i>&#93;</sup><sup id="cite&#95;ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:RS#Questionable_sources" class="mw-redirect" title="Wikipedia:RS"><span title="The material in the vicinity of this tag may rely on a promotional source. (December 2018)">promotional source?</span></a></i>&#93;</sup></li>
<li><a href="/wiki/Runtime_application_self-protection" title="Runtime application self-protection">Runtime application self-protection</a> augments existing applications to provide intrusion detection and prevention from within an application runtime.</li>
<li>Dependency scanners (also called <a href="/wiki/Software_composition_analysis" title="Software composition analysis">software composition analysis</a>) try to detect the usage of software components with known vulnerabilities. These tools can either work on-demand, e.g., during the source code build process, or periodically.</li></ul>
