<p><b>ZIP</b> is an <a href="/wiki/Archive_file_format" class="mw-redirect" title="Archive file format">archive file format</a> that supports <a href="/wiki/Lossless_compression" title="Lossless compression">lossless data compression</a>. A ZIP file may contain one or more files or directories that may have been compressed. The ZIP file format permits a number of <a href="/wiki/Data_compression" title="Data compression">compression algorithms</a>, though <a href="/wiki/DEFLATE" class="mw-redirect" title="DEFLATE">DEFLATE</a> is the most common. 
</p><p>This format was originally created in 1989 and was first implemented in <a href="/wiki/PKWARE,_Inc." class="mw-redirect" title="PKWARE, Inc.">PKWARE, Inc.</a>'s <a href="/wiki/PKZIP" title="PKZIP">PKZIP</a> utility,<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> as a replacement for the previous <a href="/wiki/ARC_(file_format)" title="ARC (file format)">ARC</a> compression format by Thom Henderson. The ZIP format was then quickly supported by many software utilities other than PKZIP. The ZIP format is since natively supported on major operating systems including <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> (where it is called "Compressed Folders"), <a href="/wiki/Apple_macOS" class="mw-redirect" title="Apple macOS">Apple macOS</a>, <a href="/wiki/Linux" title="Linux">Linux</a>, <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, <a href="/wiki/Oracle_Solaris" title="Oracle Solaris">Oracle Solaris</a>, <a href="/wiki/Android_(operating_system)" title="Android (operating system)">Android</a> and <a href="/wiki/IOS" title="IOS">iOS</a>.
</p><p>ZIP files generally use the <a href="/wiki/File_extension" class="mw-redirect" title="File extension">file extensions</a> <style data-mw-deduplicate="TemplateStyles:r886049734">.mw-parser-output .monospaced{font-family:monospace,monospace}</style><span class="monospaced">.zip</span> or <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> and the <a href="/wiki/MIME" title="MIME">MIME</a> media type <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">application/zip</code>.<sup id="cite&#95;ref-iana&#95;1-1" class="reference"><a href="#cite_note-iana-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> ZIP is used as a base file format by many programs, usually under a different name. When navigating a file system via a user interface, graphical <a href="/wiki/Icon_(computing)" title="Icon (computing)">icons</a> representing ZIP files often appear as a document or other object prominently featuring a <a href="/wiki/Zipper" title="Zipper">zipper</a>.
</p>

<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=1" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> file format was designed by <a href="/wiki/Phil_Katz" title="Phil Katz">Phil Katz</a> of <a href="/wiki/PKWARE,_Inc" class="mw-redirect" title="PKWARE, Inc">PKWARE</a> and Gary Conway of Infinity Design Concepts. The format was created after Systems Enhancement Associates (SEA) filed a <a href="/wiki/ARC_(file_format)#Lawsuits" title="ARC (file format)">lawsuit</a> against PKWARE claiming that the latter's archiving products, named PKARC, were derivatives of SEA's <a href="/wiki/ARC_(file_format)" title="ARC (file format)">ARC</a> archiving system.<sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>  The name "zip" (meaning "move at high speed") was suggested by Katz's friend, Robert Mahoney.<sup id="cite&#95;ref-BBS&#95;Documentary&#95;4-0" class="reference"><a href="#cite_note-BBS_Documentary-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>  They wanted to imply that their product would be faster than <a href="/wiki/ARC_(file_format)" title="ARC (file format)">ARC</a> and other compression formats of the time.<sup id="cite&#95;ref-BBS&#95;Documentary&#95;4-1" class="reference"><a href="#cite_note-BBS_Documentary-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>  <span class="citation-needed-content" style="padding-left:0.1em; padding-right:0.1em; color:var(--color-subtle, #54595d); border:1px solid var(--border-color-subtle, #c8ccd1);">The earliest known version of <i>.ZIP File Format Specification</i> was first published as part of <a href="/wiki/PKZIP" title="PKZIP">PKZIP</a> 0.9 package under the file APPNOTE.TXT in 1989.</span><sup class="noprint Inline-Template Template-Fact" style="margin-left:0.1em; white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (May 2018)">citation needed</span></a></i>&#93;</sup>  By distributing the zip file format within APPNOTE.TXT, compatibility with the zip file format proliferated widely on the public Internet during the 1990s.<sup id="cite&#95;ref-zipattack&#95;5-0" class="reference"><a href="#cite_note-zipattack-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup>
</p><p>PKWARE and Infinity Design Concepts made a joint press release on February 14, 1989, releasing the <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> file format into the <a href="/wiki/Public_domain" title="Public domain">public domain</a>.<sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Version_history">Version history</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=2" title="Edit section: Version history"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The .ZIP File Format Specification has its own version number, which does not necessarily correspond to the version numbers for the PKZIP tool, especially with PKZIP 6 or later. At various times, PKWARE has added preliminary features that allow PKZIP products to extract archives using advanced features, but PKZIP products that create such archives are not made available until the next major release. Other companies or organizations support the PKWARE specifications at their own pace.
</p><p>The .ZIP file format specification is formally named "APPNOTE - .ZIP File Format Specification" and it is published on the PKWARE.com website since the late 1990s.<sup id="cite&#95;ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> Several versions of the specification were not published. Specifications of some features such as <a href="/wiki/BZIP2" class="mw-redirect" title="BZIP2">BZIP2</a> compression, strong encryption specification and others were published by PKWARE a few years after their creation. The URL of the online specification was changed several times on the PKWARE website.

<div class="mw-heading mw-heading3"><h3 id="Standardization">Standardization</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=3" title="Edit section: Standardization"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In April 2010, <a href="/wiki/JTC_1" class="mw-redirect" title="JTC 1">ISO/IEC JTC 1</a> initiated a ballot to determine whether a project should be initiated to create an ISO/IEC International Standard format compatible with ZIP.<sup id="cite&#95;ref-33" class="reference"><a href="#cite_note-33"><span class="cite-bracket">&#91;</span>33<span class="cite-bracket">&#93;</span></a></sup> The proposed project, entitled <i>Document Packaging</i>, envisaged a ZIP-compatible 'minimal compressed archive format' suitable for use with a number of existing standards including <a href="/wiki/OpenDocument" title="OpenDocument">OpenDocument</a>, <a href="/wiki/Office_Open_XML" title="Office Open XML">Office Open XML</a> and <a href="/wiki/EPUB" title="EPUB">EPUB</a>. It would solve problems such as the need for a formal standard, the variety of extensions of ZIP, the undesirability of a technology used for <a href="/wiki/Open_Standard" class="mw-redirect" title="Open Standard">Open Standards</a> potentially having proprietary extensions or "submarine" patents (i.e. which could surface unexpectedly), the need for better internationalization, and a desire not to actually fragment the technology further by purporting to provide an alternative specification to the PKWARE APPNOTE document. 
</p><p>In 2015, ISO/IEC 21320-1 "Document Container File — Part 1: Core" was published which states that "Document container files are conforming Zip files", normatively referencing the PKWARE APPNOTE document. It requires the following main restrictions of the ZIP file format:<sup id="cite&#95;ref-34" class="reference"><a href="#cite_note-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li>Files in ZIP archives may only be stored uncompressed, or using the "deflate" compression (i.e. compression method may contain the value "0" - stored or "8" - deflated). The patent on the core "deflate" compression method expired in late 2010.<sup id="cite&#95;ref-35" class="reference"><a href="#cite_note-35"><span class="cite-bracket">&#91;</span>35<span class="cite-bracket">&#93;</span></a></sup></li>
<li>The encryption features are prohibited.</li>
<li>The digital signature features (from SES) are prohibited.</li>
<li>The "patched data" features (from PKPatchMaker) are prohibited.</li>
<li>Archives may not span multiple volumes or be segmented.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Design">Design</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=4" title="Edit section: Design"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> files are archives that store multiple files. ZIP allows contained files to be compressed using many different methods, as well as simply storing a file without compressing it. Each file is stored separately, allowing different files in the same archive to be compressed using different methods. Because the files in a ZIP archive are compressed individually, it is possible to extract them, or add new ones, without applying compression or decompression to the entire archive. This contrasts with the format of compressed <a href="/wiki/Tar_(file_format)" class="mw-redirect" title="Tar (file format)">tar</a> files, for which such random-access processing is not easily possible.
</p><p>A directory is placed at the end of a ZIP file. This identifies what files are in the ZIP and identifies where in the ZIP that file is located. This allows ZIP readers to load the list of files without reading the entire ZIP archive. ZIP archives can also include extra data that is not related to the ZIP archive. This allows for a ZIP archive to be made into a self-extracting archive (application that decompresses its contained data), by prepending the program code to a ZIP archive and marking the file as executable. Storing the catalog at the end also makes possible to hide a zipped file by appending it to an innocuous file, such as a GIF image file.
</p><p>The <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> format uses <a href="/wiki/CRC-32" class="mw-redirect" title="CRC-32">CRC-32</a> and includes two copies of each entry metadata to provide greater protection against data loss. The CRC-32 algorithm was contributed by David Schwaderer and can be found in his book "C Programmers Guide to NetBIOS" published by Howard W. Sams &amp; Co. Inc.<sup id="cite&#95;ref-36" class="reference"><a href="#cite_note-36"><span class="cite-bracket">&#91;</span>36<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Structure">Structure</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=5" title="Edit section: Structure"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure typeof="mw:File/Thumb"><a href="/wiki/File:ZIP-64_Internal_Layout.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/6/63/ZIP-64_Internal_Layout.svg/500px-ZIP-64_Internal_Layout.svg.png" decoding="async" width="400" height="217" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/6/63/ZIP-64_Internal_Layout.svg/960px-ZIP-64_Internal_Layout.svg.png 1.5x" data-file-width="972" data-file-height="527" /></a><figcaption>ZIP-64 Internal Layout</figcaption></figure>
<p>A ZIP file is correctly identified by the presence of an <i>end of central directory record</i> which is located at the end of the archive structure in order to allow the easy appending of new files. If the end of central directory record indicates a non-empty archive, the name of each file or directory within the archive should be specified in a <i>central directory</i> entry, along with other metadata about the entry, and an offset into the ZIP file, pointing to the actual entry data. This allows a file listing of the archive to be performed relatively quickly, as the entire archive does not have to be read to see the list of files. The entries within the ZIP file also include this information, for redundancy, in a <i>local file header</i>. Because ZIP files may be appended to, only files specified in the central directory at the end of the file are valid. Scanning a ZIP file for local file headers is invalid (except in the case of corrupted archives), as the central directory may declare that some files have been deleted and other files have been updated.
</p><p>For example, we may start with a ZIP file that contains files A, B and C. File B is then deleted and C updated. This may be achieved by just appending a new file C to the end of the original ZIP file and adding a new central directory that only lists file A and the new file C. When ZIP was first designed, transferring files by floppy disk was common, yet writing to disks was very time-consuming. If you had a large zip file, possibly spanning multiple disks, and only needed to update a few files, rather than reading and re-writing all the files, it would be substantially faster to just read the old central directory, append the new files then append an updated central directory.
</p><p>The order of the file entries in the central directory need not coincide with the order of file entries in the archive.
</p><p>Each entry stored in a ZIP archive is introduced by a <i>local file header</i> with information about the file such as the comment, file size and file name, followed by optional "extra" data fields, and then the possibly compressed, possibly encrypted file data. The "Extra" data fields are the key to the extensibility of the ZIP format. "Extra" fields are exploited to support the ZIP64 format, WinZip-compatible AES encryption, file attributes, and higher-resolution NTFS or Unix file timestamps. Other extensions are possible via the "Extra" field. ZIP tools are required by the specification to ignore Extra fields they do not recognize.
</p><p>The ZIP format uses specific 4-byte "signatures" to denote the various structures in the file. Each file entry is marked by a specific signature. The end of central directory record is indicated with its specific signature, and each entry in the central directory starts with the 4-byte <i>central file header signature</i>.
</p><p>There is no BOF or EOF marker in the ZIP specification. Conventionally the first thing in a ZIP file is a ZIP entry, which can be identified easily by its <i>local file header signature</i>. However, this is not necessarily the case, as this is not required by the ZIP specification - most notably, a self-extracting archive will begin with an executable file header.
</p><p>Tools that correctly read ZIP archives must scan for the end of central directory record signature, and then, as appropriate, the other, indicated, central directory records. They must not scan for entries from the top of the ZIP file, because (as previously mentioned in this section) only the central directory specifies where a file chunk starts and that it has not been deleted. Scanning could lead to false positives, as the format does not forbid other data to be between chunks, nor file data streams from containing such signatures. However, tools that attempt to recover data from damaged ZIP  archives will most likely scan the archive for local file header signatures; this is made more difficult by the fact that the compressed size of a file chunk may be stored after the file chunk, making sequential processing difficult.
</p><p>Most of the signatures end with the short integer 0x4b50, which is stored in <a href="/wiki/Little-endian" class="mw-redirect" title="Little-endian">little-endian</a> ordering. Viewed as an <a href="/wiki/ASCII" title="ASCII">ASCII</a> string this reads "PK", the initials of the inventor Phil Katz. Thus, when a ZIP file is viewed in a text editor the first two bytes of the file are usually "PK". (DOS, OS/2 and Windows self-extracting ZIPs have an <a href="/wiki/EXE" class="mw-redirect" title="EXE">EXE</a> before the ZIP so start with "MZ"; self-extracting ZIPs for other operating systems may similarly be preceded by executable code for extracting the archive's content on that platform.)
</p><p>The <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> specification also supports spreading archives across multiple file-system files. Originally intended for storage of large ZIP files across multiple <a href="/wiki/Floppy_disk" title="Floppy disk">floppy disks</a>, this feature is now used for sending ZIP archives in parts over email, or over other transports or removable media.
</p><p>The <a href="/wiki/File_Allocation_Table" title="File Allocation Table">FAT filesystem</a> of DOS has a timestamp resolution of only two seconds; ZIP file records mimic this. As a result, the built-in timestamp resolution of files in a ZIP archive is only two seconds, though extra fields can be used to store more precise timestamps. The ZIP format has no notion of <a href="/wiki/Time_zone" title="Time zone">time zone</a>, so timestamps are only meaningful if it is known what time zone they were created in.
</p><p>In September 2006, PKWARE released a revision of the ZIP specification providing for the storage of file names using <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a>, finally adding Unicode compatibility to ZIP.<sup id="cite&#95;ref-zip630&#95;20-1" class="reference"><a href="#cite_note-zip630-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="File_headers">File headers</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=6" title="Edit section: File headers"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>All multi-byte values in the header are stored in <a href="/wiki/Little-endian" class="mw-redirect" title="Little-endian">little-endian</a> byte order. All length fields count the length in bytes.
</p>

<p>The extra field contains a variety of optional data such as OS-specific attributes.  It is divided into records, each with at minimum a 16-bit signature and a 16-bit length. A ZIP64 local file extra field record, for example, has the signature 0x0001 and a length of 16 bytes (or more) so that two 64-bit values (the uncompressed and compressed sizes) may follow. Another common local file extension is 0x5455 (or "UT") which contains 32-bit UTC UNIX timestamps.
</p><p>This is immediately followed by the compressed data.
</p>

<div class="mw-heading mw-heading3"><h3 id="Encryption">Encryption</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=12" title="Edit section: Encryption"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>ZIP supports a simple <a href="/wiki/Password" title="Password">password</a>-based <a href="/wiki/Symmetric-key_algorithm" title="Symmetric-key algorithm">symmetric encryption</a> system generally known as ZipCrypto. It is documented in the ZIP specification, and known to be seriously flawed. In particular, it is vulnerable to <a href="/wiki/Known-plaintext_attack" title="Known-plaintext attack">known-plaintext attacks</a>, which are in some cases made worse by poor implementations of <a href="/wiki/Random-number_generator" class="mw-redirect" title="Random-number generator">random-number generators</a>.<sup id="cite&#95;ref-zipattack&#95;5-1" class="reference"><a href="#cite_note-zipattack-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> Computers running under native <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> without third-party archivers can open, but not create, ZIP files encrypted with ZipCrypto, but cannot extract the contents of files using different encryption.<sup id="cite&#95;ref-41" class="reference"><a href="#cite_note-41"><span class="cite-bracket">&#91;</span>41<span class="cite-bracket">&#93;</span></a></sup>
</p><p>New features including new <a href="/wiki/Data_compression" title="Data compression">compression</a> and <a href="/wiki/Encryption" title="Encryption">encryption</a> (e.g. <a href="/wiki/Advanced_Encryption_Standard" title="Advanced Encryption Standard">AES</a>) methods have been documented in the ZIP File Format Specification since version 5.2. A <a href="/wiki/WinZip" title="WinZip">WinZip</a>-developed AES-based open standard ("AE-x" in APPNOTE) is used also by <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a> and <a href="/w/index.php?title=Xceed_(software_company)&amp;action=edit&amp;redlink=1" class="new" title="Xceed (software company) (page does not exist)">Xceed</a>, but some vendors use other formats.<sup id="cite&#95;ref-42" class="reference"><a href="#cite_note-42"><span class="cite-bracket">&#91;</span>42<span class="cite-bracket">&#93;</span></a></sup> PKWARE SecureZIP (SES, proprietary) also supports RC2, RC4, DES, Triple DES encryption methods, Digital Certificate-based encryption and authentication (<a href="/wiki/X.509" title="X.509">X.509</a>), and archive header encryption. It is, however, patented (see <a href="#Strong_encryption_controversy">§&#160;Strong encryption controversy</a>).<sup id="cite&#95;ref-pkware&#95;43-0" class="reference"><a href="#cite_note-pkware-43"><span class="cite-bracket">&#91;</span>43<span class="cite-bracket">&#93;</span></a></sup>
</p><p><a href="/wiki/File_name" class="mw-redirect" title="File name">File name</a> <a href="/wiki/Encryption" title="Encryption">encryption</a> is introduced in .ZIP File Format Specification 6.2, which encrypts metadata stored in Central Directory portion of an archive, but Local Header sections remain unencrypted. A compliant archiver can falsify the Local Header data when using Central Directory Encryption. As of version 6.2 of the specification, the Compression Method and Compressed Size fields within Local Header are not yet masked.
</p>
<div class="mw-heading mw-heading3"><h3 id="ZIP64">ZIP64</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=13" title="Edit section: ZIP64"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The original <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> format had a 4&#160;<a href="/wiki/GiB" class="mw-redirect" title="GiB">GiB</a> (<span class="texhtml">2<sup>32</sup></span> bytes) limit on various things (uncompressed size of a file, compressed size of a file, and total size of the archive), as well as a limit of 65,535 (<span class="texhtml">2<sup>16</sup> − 1</span>) entries in a ZIP archive. In version 4.5 of the specification (which is not the same as v4.5 of any particular tool), PKWARE introduced the "ZIP64" format extensions to get around these limitations, increasing the limits to 16&#160;<a href="/wiki/EiB" class="mw-redirect" title="EiB">EiB</a> (<span class="texhtml">2<sup>64</sup></span> bytes). In essence, it uses a "normal" central directory entry for a file, followed by an optional "zip64" directory entry, which has the larger fields.<sup id="cite&#95;ref-44" class="reference"><a href="#cite_note-44"><span class="cite-bracket">&#91;</span>44<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The format of the Local file header (LOC) and Central directory file header (CDFH) are the same in ZIP and ZIP64. However, ZIP64 specifies an extra field that may be added to those records at the discretion of the compressor, whose purpose is to store values that do not fit in the classic LOC or CDFH records. To signal that the actual values are stored in ZIP64 extra fields, they are set to 0xFFFF or 0xFFFFFFFF in the corresponding LOC or CDFH record. If one entry does not fit into the classic LOC or CDFH record, only that entry is required to be moved into a ZIP64 extra field. The other entries may stay in the classic record. Therefore, not all entries shown in the following table might be stored in a ZIP64 extra field. However, if they appear, their order must be as shown in the table.

<div class="mw-heading mw-heading3"><h3 id="Combination_with_other_file_formats">Combination with other file formats</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=14" title="Edit section: Combination with other file formats"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> file format allows for a comment containing up to 65,535 (<span class="texhtml">2<sup>16</sup> − 1</span>) bytes of data to occur at the end of the file after the central directory.<sup id="cite&#95;ref-appnote&#95;37-8" class="reference"><a href="#cite_note-appnote-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> Also, because the central directory specifies the offset of each file in the archive with respect to the start, it is possible for the first file entry to start at an offset other than zero, although some tools might not process archive files that do not start with a file entry at offset zero. The program <a href="/wiki/Gzip" title="Gzip">gzip</a>, for example, happens to be able to extract an entry from a .ZIP file if it is at offset zero.
</p><p>This allows arbitrary data to occur in the file both before and after the ZIP archive data, and for the archive to still be read by a ZIP application. A side-effect of this is that it is possible to author a file that is both a working ZIP archive and another format, provided that the other format tolerates arbitrary data at its end, beginning, or middle. <a href="/wiki/Self-extracting_archives" class="mw-redirect" title="Self-extracting archives">Self-extracting archives</a> (SFX), of the form supported by WinZip, take advantage of this, in that they are executable (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.exe</span>) files that conform to the PKZIP AppNote.txt specification, and can be read by compliant zip tools or libraries.
</p><p>This property of the <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> format, and of the <a href="/wiki/JAR_(file_format)" title="JAR (file format)">JAR</a> format which is a variant of ZIP, can be exploited to hide rogue content (such as harmful Java classes) inside a seemingly harmless file, such as a GIF image uploaded to the web. This so-called <a href="/wiki/Polyglot_(computing)#GIFAR_attack" title="Polyglot (computing)">GIFAR</a> exploit has been demonstrated as an effective attack against web applications such as Facebook.<sup id="cite&#95;ref-51" class="reference"><a href="#cite_note-51"><span class="cite-bracket">&#91;</span>51<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Limits">Limits</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=15" title="Edit section: Limits"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The minimum size of a <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> file is 22 bytes. Such an <i>empty zip file</i> contains only an End of Central Directory Record (EOCD): <code>50 4B 05 06 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00</code>
</p><p>The maximum size for both the archive file and the individual files inside it is 4,294,967,295 bytes (<span class="texhtml">2<sup>32</sup> − 1</span> bytes, or 4&#160;GiB minus 1 byte) for standard ZIP. For ZIP64, the maximum size is 18,446,744,073,709,551,615 bytes (<span class="texhtml">2<sup>64</sup> − 1</span> bytes, or 16&#160;EiB minus 1 byte).<sup id="cite&#95;ref-ziplimit&#95;52-0" class="reference"><a href="#cite_note-ziplimit-52"><span class="cite-bracket">&#91;</span>52<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Open_extensions">Open extensions</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=16" title="Edit section: Open extensions"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading4"><h4 id="Seek-optimized_(SOZip)_profile"><span id="Seek-optimized_.28SOZip.29_profile"></span>Seek-optimized (SOZip) profile</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=17" title="Edit section: Seek-optimized (SOZip) profile"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>A Seek-Optimized ZIP file (SOZip) profile<sup id="cite&#95;ref-53" class="reference"><a href="#cite_note-53"><span class="cite-bracket">&#91;</span>53<span class="cite-bracket">&#93;</span></a></sup> has been proposed for the ZIP format. Such file contains one or several Deflate-compressed files that are organized and annotated such that a SOZip-aware reader can perform very fast random access (seek) within a compressed file. SOZip makes it possible to access large compressed files directly from a .zip file without prior decompression. It combines the use of ZLib block flushes issued at regular interval with a hidden index file mapping offsets of the uncompressed file to offsets in the compressed stream. ZIP readers that are not aware of that extension can read a SOZip-enabled file normally and ignore the extended features that support efficient seek capability.
</p>
<div class="mw-heading mw-heading3"><h3 id="Proprietary_extensions">Proprietary extensions</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=18" title="Edit section: Proprietary extensions"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading4"><h4 id="Extra_field">Extra field</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=19" title="Edit section: Extra field"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> file format includes an extra field facility within file headers, which can be used to store extra data not defined by existing ZIP specifications, and which allow compliant archivers that do not recognize the fields to safely skip them. Header IDs 0–31 are reserved for use by PKWARE. The remaining IDs can be used by third-party vendors for proprietary usage.
</p>
<div class="mw-heading mw-heading4"><h4 id="Strong_encryption_controversy">Strong encryption controversy</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=20" title="Edit section: Strong encryption controversy"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>When <a href="/wiki/WinZip" title="WinZip">WinZip</a> 9.0 public beta was released in 2003, WinZip introduced its own <a href="/wiki/AES-256" class="mw-redirect" title="AES-256">AES-256</a> encryption, using a different file format, along with the documentation for the new specification.<sup id="cite&#95;ref-54" class="reference"><a href="#cite_note-54"><span class="cite-bracket">&#91;</span>54<span class="cite-bracket">&#93;</span></a></sup> The encryption standards themselves were not <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary</a>, but PKWARE had not updated APPNOTE.TXT to include Strong Encryption Specification (SES) since 2001, which had been used by PKZIP versions 5.0 and 6.0. WinZip technical consultant Kevin Kearney and <a href="/wiki/StuffIt" title="StuffIt">StuffIt</a> product manager Mathew Covington accused PKWARE of withholding SES, but PKZIP chief technology officer Jim Peterson claimed that certificate-based encryption was still incomplete.
</p><p>In another controversial move, PKWare applied for a patent on 16 July 2003 describing a method for combining ZIP and strong encryption to create a secure file.<sup id="cite&#95;ref-55" class="reference"><a href="#cite_note-55"><span class="cite-bracket">&#91;</span>55<span class="cite-bracket">&#93;</span></a></sup>
</p><p>In the end, PKWARE and WinZip agreed to support each other's products. On 21 January 2004, PKWARE announced the support of WinZip-based AES compression format.<sup id="cite&#95;ref-56" class="reference"><a href="#cite_note-56"><span class="cite-bracket">&#91;</span>56<span class="cite-bracket">&#93;</span></a></sup> In a later version of WinZip beta, it was able to support SES-based ZIP files.<sup id="cite&#95;ref-57" class="reference"><a href="#cite_note-57"><span class="cite-bracket">&#91;</span>57<span class="cite-bracket">&#93;</span></a></sup> PKWARE eventually released version 5.2 of the .ZIP File Format Specification to the public, which documented SES.  The <a href="/wiki/Free_Software" class="mw-redirect" title="Free Software">Free Software</a> project <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a> also supports AES, but not SES in ZIP files (as does its <a href="/wiki/POSIX" title="POSIX">POSIX</a> <a href="/wiki/Porting" title="Porting">port</a> <a href="/wiki/P7zip" class="mw-redirect" title="P7zip">p7zip</a>).
</p><p>When using AES encryption under WinZip, the compression method is always set to 99, with the actual compression method stored in an AES extra data field.<sup id="cite&#95;ref-58" class="reference"><a href="#cite_note-58"><span class="cite-bracket">&#91;</span>58<span class="cite-bracket">&#93;</span></a></sup> In contrast, Strong Encryption Specification stores the compression method in the basic file header segment of Local Header and Central Directory, unless Central Directory Encryption is used to mask/encrypt metadata.
</p>
<div class="mw-heading mw-heading2"><h2 id="Implementation">Implementation</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=21" title="Edit section: Implementation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>There are numerous <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> tools available, and numerous <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> libraries for various programming environments; licenses used include <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary</a> and <a href="/wiki/Free_software" title="Free software">free software</a>. <a href="/wiki/WinZip" title="WinZip">WinZip</a>, <a href="/wiki/WinRAR" title="WinRAR">WinRAR</a>, <a href="/wiki/Info-ZIP" title="Info-ZIP">Info-ZIP</a>, <a href="/wiki/ZipGenius" title="ZipGenius">ZipGenius</a>, <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a>, <a href="/wiki/PeaZip" title="PeaZip">PeaZip</a> and <a href="/wiki/B1_Free_Archiver" title="B1 Free Archiver">B1 Free Archiver</a> are well-known <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> tools, available on various platforms. Some of those tools have library or programmatic interfaces.
</p><p>Some development libraries licensed under open source agreement are <a href="/wiki/Libzip" title="Libzip">libzip</a>, <a href="/wiki/Libarchive" title="Libarchive">libarchive</a>, and <a href="/wiki/Info-ZIP" title="Info-ZIP">Info-ZIP</a>. For Java: <a href="/wiki/Java_Platform,_Standard_Edition" title="Java Platform, Standard Edition">Java Platform, Standard Edition</a> contains the package "java.util.zip" to handle standard <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> files; the Zip64File library specifically supports large files (larger than 4&#160;GiB) and treats <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> files using random access; and the <a href="/wiki/Apache_Ant" title="Apache Ant">Apache Ant</a> tool contains a more complete implementation released under the <a href="/wiki/Apache_Software_License" class="mw-redirect" title="Apache Software License">Apache Software License</a>.
</p><p>The <a href="/wiki/Info-ZIP" title="Info-ZIP">Info-ZIP</a> implementations of the <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> format adds support for Unix filesystem features, such as user and group IDs, file permissions, and support for symbolic links. The <a href="/wiki/Apache_Ant" title="Apache Ant">Apache Ant</a> implementation is aware of these to the extent that it can create files with predefined Unix permissions. The Info-ZIP implementations also know how to use the error correction capabilities built into the <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> compression format. Some programs do not, and will fail on a file that has errors.
</p><p>The Info-ZIP Windows tools also support <a href="/wiki/NTFS" title="NTFS">NTFS</a> <a href="/wiki/Filesystem" class="mw-redirect" title="Filesystem">filesystem</a> permissions, and will make an attempt to translate from NTFS permissions to Unix permissions or vice versa when extracting files. This can result in potentially unintended combinations, e.g. <a href="/wiki/.exe" class="mw-redirect" title=".exe">.exe</a> files being created on NTFS volumes with executable permission denied.
</p><p>Versions of Microsoft Windows have included support for <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> compression in Explorer since the <a href="/wiki/Microsoft_Plus!" title="Microsoft Plus!">Microsoft Plus!</a> pack was released for <a href="/wiki/Windows_98" title="Windows 98">Windows 98</a>. Microsoft calls this feature "Compressed Folders". Native support was added as of the year 2000 in <a href="/wiki/Windows_ME" class="mw-redirect" title="Windows ME">Windows ME</a>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (October 2023)">citation needed</span></a></i>&#93;</sup> Not all <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">.ZIP</span> features are supported by the Windows Compressed Folders capability. For example, encryption is not supported in Windows 10 Home edition,<sup id="cite&#95;ref-59" class="reference"><a href="#cite_note-59"><span class="cite-bracket">&#91;</span>59<span class="cite-bracket">&#93;</span></a></sup> although it can decrypt. Unicode entry encoding is not supported until <a href="/wiki/Windows_7" title="Windows 7">Windows 7</a>, while split and spanned archives are not readable or writable by the Compressed Folders feature, nor is AES Encryption supported.<sup id="cite&#95;ref-60" class="reference"><a href="#cite_note-60"><span class="cite-bracket">&#91;</span>60<span class="cite-bracket">&#93;</span></a></sup> Windows .zip support stemmed from an acquisition of "VisualZip" written by <a href="/wiki/Dave_Plummer" title="Dave Plummer">Dave Plummer</a>.<sup id="cite&#95;ref-61" class="reference"><a href="#cite_note-61"><span class="cite-bracket">&#91;</span>61<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-62" class="reference"><a href="#cite_note-62"><span class="cite-bracket">&#91;</span>62<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-63" class="reference"><a href="#cite_note-63"><span class="cite-bracket">&#91;</span>63<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Apple has included built-in ZIP support in <a href="/wiki/MacOS" title="MacOS">Mac&#160;OS&#160;X</a> 10.3 (via BOMArchiveHelper, now <a href="/wiki/Archive_Utility" class="mw-redirect" title="Archive Utility">Archive Utility</a>) and later. Most <a href="/wiki/Category:Free_software_operating_systems" title="Category:Free software operating systems">free operating systems</a> have built in support for ZIP in similar manners to Windows and macOS.
</p><p><a href="/wiki/OpenDocument" title="OpenDocument">OpenDocument</a> Format (ODF) started using the zip archive format in 2005, ODF is an open format for office documents of all types, this is the default file format used in <a href="/wiki/Collabora_Online" title="Collabora Online">Collabora Online</a>, <a href="/wiki/LibreOffice" title="LibreOffice">LibreOffice</a> and others.<sup id="cite&#95;ref-64" class="reference"><a href="#cite_note-64"><span class="cite-bracket">&#91;</span>64<span class="cite-bracket">&#93;</span></a></sup>  Microsoft Office started using the zip archive format in 2006 for their <a href="/wiki/Office_Open_XML" title="Office Open XML">Office Open XML</a> .docx, .xlsx, .pptx, etc. files, which became the default file format with <a href="/wiki/Microsoft_Office_2007" title="Microsoft Office 2007">Microsoft Office 2007</a>.
</p>
<div class="mw-heading mw-heading3"><h3 id="Internationalization_issues">Internationalization issues</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=22" title="Edit section: Internationalization issues"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Versions of the format prior to 6.3.0 did not support storing file names in <a href="/wiki/Unicode" title="Unicode">Unicode</a>.<sup id="cite&#95;ref-:0&#95;65-0" class="reference"><a href="#cite_note-:0-65"><span class="cite-bracket">&#91;</span>65<span class="cite-bracket">&#93;</span></a></sup> According to the standard,<sup id="cite&#95;ref-:0&#95;65-1" class="reference"><a href="#cite_note-:0-65"><span class="cite-bracket">&#91;</span>65<span class="cite-bracket">&#93;</span></a></sup> file names should be stored in the <a href="/wiki/CP437" class="mw-redirect" title="CP437">CP437</a> encoding, which is standard for the <a href="/wiki/IBM_PC" class="mw-redirect" title="IBM PC">IBM PC</a>,<sup id="cite&#95;ref-:0&#95;65-2" class="reference"><a href="#cite_note-:0-65"><span class="cite-bracket">&#91;</span>65<span class="cite-bracket">&#93;</span></a></sup> but in practice, <a href="/wiki/DOS" title="DOS">DOS</a> archivers used the system's installed <a href="/wiki/Character_encoding" title="Character encoding">character encoding</a>. The built-in archiver of Windows up to 11 also used the DOS encoding corresponding to the selected system language for backward compatibility when creating archives. Subsequently, the standard was updated to include two options for storing file names in Unicode: 1) when the 11th bit in the General purpose bit flag field is set, the file name in the "File name" field of the header should be considered as <a href="/wiki/UTF-8" title="UTF-8">UTF-8</a> rather than a single-byte encoding, and 2) the Unicode Path Extra Field was added to store the file name in UTF-8 encoding.<sup id="cite&#95;ref-:0&#95;65-3" class="reference"><a href="#cite_note-:0-65"><span class="cite-bracket">&#91;</span>65<span class="cite-bracket">&#93;</span></a></sup> Some versions of archivers on the Windows platform have also used ANSI encoding in the past. Thus, to correctly extract files with names containing non-English characters, it is necessary:<sup id="cite&#95;ref-auto&#95;66-0" class="reference"><a href="#cite_note-auto-66"><span class="cite-bracket">&#91;</span>66<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ol><li>Check for the presence of the Unicode Path Extra Field, and if it exists, use the filename from it, encoded in UTF-8.</li>
<li>Check for the presence of flag 11 in the General purpose bit flag field, and if it is set, consider the filename encoding in the "File name" field to be UTF-8.</li>
<li>If the "packing OS" field contains the value 11 (NTFS, Windows), and the "version of the packer" field value is greater than or equal to 20, consider the filename encoding in the "File name" field to be the ANSI (Windows) encoding corresponding to the system locale if one can be determined; otherwise, use CP437.</li>
<li>If the "packing OS" field contains the value 0 (FAT, DOS), and the "version of the packer" field value is between 25 and 40 inclusive, consider the filename encoding in the local header's "File name" field to be ANSI (Windows) encoding, and in the central header's "File name" field to be OEM (DOS) encoding, corresponding to the system locale if one can be determined; otherwise, use CP437.</li>
<li>In other cases, if the "OS packing" field contains the value 0 (FAT, DOS), 6 (HPFS, OS/2), or 11 (NTFS, Windows), consider the filename encoding in the "File name" field to be OEM (DOS) encoding, corresponding to the system locale if one can be determined; otherwise, use CP437.</li>
<li>In all other cases, consider the filename encoding in the "File name" field to be the system encoding of operating system unpacker is running on.</li></ol>
<p>Some implementations of zip unpackers did not implement this algorithm or only partially implemented it, as a result, when viewing the contents of an archive or extracting it, users saw a chaotic set of characters, known as "<a href="/wiki/Mojibake" title="Mojibake">mojibake</a>", instead of letters of the national alphabet. In 2016, this problem was solved in the <a href="/wiki/FAR_Manager#Linux,_MacOS_and_BSD_version" class="mw-redirect" title="FAR Manager">far2l</a> file and archive manager for Linux, BSD and Mac.<sup id="cite&#95;ref-67" class="reference"><a href="#cite_note-67"><span class="cite-bracket">&#91;</span>67<span class="cite-bracket">&#93;</span></a></sup> In 2024, similar solution was added<sup id="cite&#95;ref-:1&#95;68-0" class="reference"><a href="#cite_note-:1-68"><span class="cite-bracket">&#91;</span>68<span class="cite-bracket">&#93;</span></a></sup> to the version of 7zip used in the <a href="/wiki/Debian" title="Debian">Debian</a> distribution and its derivatives, and to the version of unzip used in the <a href="/wiki/Ubuntu" title="Ubuntu">Ubuntu</a> distribution and its derivatives.<sup id="cite&#95;ref-auto&#95;66-1" class="reference"><a href="#cite_note-auto-66"><span class="cite-bracket">&#91;</span>66<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Legacy">Legacy</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=23" title="Edit section: Legacy"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>There are numerous other standards and formats using "zip" as part of their name. For example, zip is distinct from <a href="/wiki/Gzip" title="Gzip">gzip</a>, and the latter is defined in <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc1952">1952</a>. Both zip and gzip primarily use the <a href="/wiki/DEFLATE" class="mw-redirect" title="DEFLATE">DEFLATE</a> algorithm for compression. Likewise, the <a href="/wiki/ZLIB" class="mw-redirect" title="ZLIB">ZLIB</a> format (IETF <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc1950">1950</a>) also uses the DEFLATE compression algorithm, but specifies different headers for error and consistency checking. Other common, similarly named formats and programs with different native formats include <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a>, <a href="/wiki/Bzip2" title="Bzip2">bzip2</a>, and <a href="/wiki/Rzip" title="Rzip">rzip</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="Concerns">Concerns</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=ZIP_(file_format)&amp;action=edit&amp;section=24" title="Edit section: Concerns"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The theoretical maximum compression factor for a raw DEFLATE stream is about 1032 to one,<sup id="cite&#95;ref-69" class="reference"><a href="#cite_note-69"><span class="cite-bracket">&#91;</span>69<span class="cite-bracket">&#93;</span></a></sup> but by exploiting the ZIP format in unintended ways, ZIP archives with compression ratios of billions to one can be constructed. These <a href="/wiki/Zip_bomb" title="Zip bomb">zip bombs</a> unzip to extremely large sizes, overwhelming the capacity of the computer they are decompressed on.<sup id="cite&#95;ref-70" class="reference"><a href="#cite_note-70"><span class="cite-bracket">&#91;</span>70<span class="cite-bracket">&#93;</span></a></sup>
</p>
