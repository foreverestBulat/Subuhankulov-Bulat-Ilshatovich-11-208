<p><b>Breadth-first search</b> (<b>BFS</b>) is an <a href="/wiki/Algorithm" title="Algorithm">algorithm</a> for searching a <a href="/wiki/Tree_(data_structure)" class="mw-redirect" title="Tree (data structure)">tree</a> data structure for a node that satisfies a given property. It starts at the <a href="/wiki/Tree_(data_structure)#Terminology" class="mw-redirect" title="Tree (data structure)">tree root</a> and explores all nodes at the present <a href="/wiki/Tree_(data_structure)#Terminology" class="mw-redirect" title="Tree (data structure)">depth</a> prior to moving on to the nodes at the next depth level. Extra memory, usually a <a href="/wiki/Queue_(data_structure)" class="mw-redirect" title="Queue (data structure)">queue</a>, is needed to keep track of the child nodes that were encountered but not yet explored. 
</p><p>For example, in a <a href="/wiki/Chess_endgame" title="Chess endgame">chess endgame</a>, a <a href="/wiki/Chess_engine" title="Chess engine">chess engine</a> may build the <a href="/wiki/Game_tree" title="Game tree">game tree</a> from the current position by applying all possible moves and use breadth-first search to find a winning position for White. Implicit trees (such as game trees or other problem-solving trees) may be of infinite size; breadth-first search is guaranteed to find a solution node<sup id="cite&#95;ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> if one exists.
</p><p>In contrast, (plain) <a href="/wiki/Depth-first_search" title="Depth-first search">depth-first search</a> (DFS), which explores the node branch as far as possible before backtracking and expanding other nodes,<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> may get lost in an infinite branch and never make it to the solution node. <a href="/wiki/Iterative_deepening_depth-first_search" title="Iterative deepening depth-first search">Iterative deepening depth-first search</a> avoids the latter drawback at the price of exploring the tree's top parts over and over again. On the other hand, both depth-first algorithms typically require far less extra memory than breadth-first search.<sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Breadth-first search can be generalized to both <a href="/wiki/Undirected_graph" class="mw-redirect" title="Undirected graph">undirected graphs</a> and <a href="/wiki/Directed_graph" title="Directed graph">directed graphs</a> with a given start node (sometimes referred to as a 'search key').<sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> In <a href="/wiki/State_space_search" class="mw-redirect" title="State space search">state space search</a> in <a href="/wiki/Artificial_intelligence" title="Artificial intelligence">artificial intelligence</a>, repeated searches of vertices are often allowed, while in theoretical analysis of algorithms based on breadth-first search, precautions are typically taken to prevent repetitions.
</p><p>BFS and its application in finding <a href="/wiki/Connected_component_(graph_theory)" class="mw-redirect" title="Connected component (graph theory)">connected components</a> of graphs were invented in 1945 by <a href="/wiki/Konrad_Zuse" title="Konrad Zuse">Konrad Zuse</a>, in his (rejected) Ph.D. thesis on the <a href="/wiki/Plankalk%C3%BCl" title="Plankalkül">Plankalkül</a> programming language, but this was not published until 1972.<sup id="cite&#95;ref-:0&#95;5-0" class="reference"><a href="#cite_note-:0-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> It was reinvented in 1959 by <a href="/wiki/Edward_F._Moore" title="Edward F. Moore">Edward F. Moore</a>, who used it to find the shortest path out of a maze,<sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-skiena&#95;7-0" class="reference"><a href="#cite_note-skiena-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> and later developed by C. Y. Lee into a <a href="/wiki/Routing_(electronic_design_automation)" title="Routing (electronic design automation)">wire routing</a> algorithm (published in 1961).<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup>
</p>

<div class="mw-heading mw-heading2"><h2 id="Pseudocode">Pseudocode</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Breadth-first_search&amp;action=edit&amp;section=1" title="Edit section: Pseudocode"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><b>Input</b>: A graph <span class="texhtml mvar" style="font-style:italic;">G</span> and a starting vertex <span class="texhtml mvar" style="font-style:italic;">root</span> of <span class="texhtml mvar" style="font-style:italic;">G</span>
</p><p><b>Output</b>: Goal state. The <i>parent</i> links trace the shortest path back to <span class="texhtml mvar" style="font-style:italic;">root</span><sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup>
</p>
<pre> 1  <b>procedure</b> BFS(<i>G</i>, <i>root</i>) <b>is</b>
 2      let <i>Q</i> be a queue
 3      label <i>root</i> as explored
 4      <i>Q</i>.enqueue(<i>root</i>)
 5      <b>while</b> <i>Q</i> is not empty <b>do</b>
 6          <i>v</i>&#160;:= <i>Q</i>.dequeue()
 7          <b>if</b> <i>v</i> is the goal <b>then</b>
 8              <b>return</b> <i>v</i>
 9          <b>for all</b> edges from <i>v</i> to <i>w</i> <b>in</b> <i>G</i>.adjacentEdges(<i>v</i>) <b>do</b>
10              <b>if</b> <i>w</i> is not labeled as explored <b>then</b>
11                  label <i>w</i> as explored
12                  <i>w</i>.parent&#160;:= <i>v</i>
13                  <i>Q</i>.enqueue(<i>w</i>)
</pre>

<p>This non-recursive implementation is similar to the non-recursive implementation of <a href="/wiki/Depth-first_search" title="Depth-first search">depth-first search</a>, but differs from it in two ways:
</p>
<ol><li>it uses a <a href="/wiki/Queue_(abstract_data_type)" title="Queue (abstract data type)">queue</a> (<a href="/wiki/First_In_First_Out" class="mw-redirect" title="First In First Out">First In First Out</a>) instead of a <a href="/wiki/Stack_(abstract_data_type)" title="Stack (abstract data type)">stack</a> (Last In First Out) and</li>
<li>it checks whether a vertex has been explored before enqueueing the vertex rather than delaying this check until the vertex is dequeued from the queue.</li></ol>
<p>If <span class="texhtml mvar" style="font-style:italic;">G</span> is a <a href="/wiki/Tree_(data_structure)" class="mw-redirect" title="Tree (data structure)">tree</a>, replacing the queue of this breadth-first search algorithm with a stack will yield a depth-first search algorithm. For general graphs, replacing the stack of the iterative depth-first search implementation with a queue would also produce a breadth-first search algorithm, although a somewhat nonstandard one.<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The <i>Q</i> queue contains the frontier along which the algorithm is currently searching.
</p><p>Nodes can be labelled as explored by storing them in a set, or by an attribute on each node, depending on the implementation.
</p><p>Note that the word <i>node</i> is usually interchangeable with the word <i>vertex</i>.
</p><p>The <i>parent</i> attribute of each node is useful for accessing the nodes in a shortest path, for example by backtracking from the destination node up to the starting node, once the BFS has been run, and the predecessors nodes have been set.
</p><p>Breadth-first search produces a <i>breadth-first tree</i> which is shown in the example below.
</p>

<div class="mw-heading mw-heading3"><h3 id="Completeness">Completeness</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Breadth-first_search&amp;action=edit&amp;section=6" title="Edit section: Completeness"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In the analysis of algorithms, the input to breadth-first search is assumed to be a finite graph, represented as an <a href="/wiki/Adjacency_list" title="Adjacency list">adjacency list</a>, <a href="/wiki/Adjacency_matrix" title="Adjacency matrix">adjacency matrix</a>, or similar representation. However, in the application of graph traversal methods in <a href="/wiki/Artificial_intelligence" title="Artificial intelligence">artificial intelligence</a> the input may be an <a href="/wiki/Implicit_graph" title="Implicit graph">implicit representation</a> of an infinite graph. In this context, a search method is described as being complete if it is guaranteed to find a goal state if one exists. Breadth-first search is complete, but depth-first search is not. When applied to infinite graphs represented implicitly, breadth-first search will eventually find the goal state, but depth first search may get lost in parts of the graph that have no goal state and never return.<sup id="cite&#95;ref-coppin&#95;13-0" class="reference"><a href="#cite_note-coppin-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup>
</p>

<div class="mw-heading mw-heading2"><h2 id="Applications">Applications</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Breadth-first_search&amp;action=edit&amp;section=8" title="Edit section: Applications"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Breadth-first search can be used to solve many problems in graph theory, for example:
</p>
<ul><li>Copying <a href="/wiki/Garbage_collection_(computer_science)" title="Garbage collection (computer science)">garbage collection</a>, <a href="/wiki/Cheney%27s_algorithm" title="Cheney&#39;s algorithm">Cheney's algorithm</a></li>
<li>Finding the <a href="/wiki/Shortest_path" class="mw-redirect" title="Shortest path">shortest path</a> between two nodes <i>u</i> and <i>v</i>, with path length measured by number of edges (an advantage over <a href="/wiki/Depth-first_search" title="Depth-first search">depth-first search</a>)<sup id="cite&#95;ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup></li>
<li><a href="/wiki/Cuthill%E2%80%93McKee_algorithm" title="Cuthill–McKee algorithm">(Reverse) Cuthill–McKee</a> mesh numbering</li>
<li><a href="/wiki/Ford%E2%80%93Fulkerson_algorithm" title="Ford–Fulkerson algorithm">Ford–Fulkerson method</a> for computing the <a href="/wiki/Maximum_flow_problem" title="Maximum flow problem">maximum flow</a> in a <a href="/wiki/Flow_network" title="Flow network">flow network</a></li>
<li>Serialization/Deserialization of a binary tree vs serialization in sorted order, allows the tree to be re-constructed in an efficient manner.</li>
<li>Construction of the <i>failure function</i> of the <a href="/wiki/Aho-Corasick" class="mw-redirect" title="Aho-Corasick">Aho-Corasick</a> pattern matcher.</li>
<li>Testing <a href="/wiki/Bipartite_graph#Testing_bipartiteness" title="Bipartite graph">bipartiteness of a graph</a>.</li>
<li>Implementing parallel algorithms for computing a graph's transitive closure.<sup id="cite&#95;ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup></li></ul>
