</div><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Domain at the highest level of the DNS hierarchy</div>
<style data-mw-deduplicate="TemplateStyles:r1320445320">.mw-parser-output .hatnote{font-style:italic}.mw-parser-output div.hatnote{padding-left:1.6em;margin-bottom:0.5em}.mw-parser-output .hatnote i{font-style:normal}.mw-parser-output .hatnote+link+.hatnote{margin-top:-0.5em}@media print{body.ns-0 .mw-parser-output .hatnote{display:none!important}}</style><div role="note" class="hatnote navigation-not-searchable">"TLD" and "TLDN" redirect here. For other uses, see <a href="/wiki/TLD_(disambiguation)" class="mw-disambig" title="TLD (disambiguation)">TLD (disambiguation)</a>.</div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">For a list of all top-level domains, see <a href="/wiki/List_of_Internet_top-level_domains" title="List of Internet top-level domains">List of Internet top-level domains</a>.</div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Country_code_top-level_domain" title="Country code top-level domain">Country code top-level domain</a></div>
<p class="mw-empty-elt">
</p><p>A <b>top-level domain</b> (<b>TLD</b>) is one of the <a href="/wiki/Domain_name" title="Domain name">domains</a> at the highest level in the hierarchical <a href="/wiki/Domain_Name_System" title="Domain Name System">Domain Name System</a> of the <a href="/wiki/Internet" title="Internet">Internet</a> after the root domain.<sup id="cite&#95;ref-rfc1591&#95;1-0" class="reference"><a href="#cite_note-rfc1591-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> The top-level domain names are installed in the <a href="/wiki/DNS_root_zone" title="DNS root zone">root zone</a> of the name space. For all domains in lower levels, it is the last part of the <a href="/wiki/Domain_name" title="Domain name">domain name</a>, that is, the last non-empty label of a <a href="/wiki/Fully_qualified_domain_name" title="Fully qualified domain name">fully qualified domain name</a>. For example, in the domain name <a href="/wiki/Example.com" title="Example.com">www.example.com</a>, the top-level domain is <a href="/wiki/.com" title=".com">.com</a>. Responsibility for management of most top-level domains is delegated to specific organizations by the <a href="/wiki/ICANN" title="ICANN">ICANN</a>, an Internet multi-stakeholder community, which operates the <a href="/wiki/Internet_Assigned_Numbers_Authority" title="Internet Assigned Numbers Authority">Internet Assigned Numbers Authority</a> (IANA), and is in charge of maintaining the <a href="/wiki/DNS_root_zone" title="DNS root zone">DNS root zone</a>.
</p>

<p>Originally, the top-level domain space was organized into three main groups: <i>Countries</i>, <i>Categories</i>, and <i>Multiorganizations</i>.<sup id="cite&#95;ref-rfc920&#95;2-0" class="reference"><a href="#cite_note-rfc920-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> An additional <i>temporary</i> group consisted of only the initial DNS domain, <a href="/wiki/.arpa" title=".arpa">.arpa</a>,<sup id="cite&#95;ref-rfc921&#95;3-0" class="reference"><a href="#cite_note-rfc921-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> and was intended for transitional purposes toward the stabilization of the domain name system.
</p>
<div class="mw-heading mw-heading2"><h2 id="Types">Types</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=2" title="Edit section: Types"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>As of 2015,<sup class="plainlinks noexcerpt noprint asof-tag update" style="display:none;"><a class="external text" href="https://en.wikipedia.org/w/index.php?title=Top-level_domain&amp;action=edit">&#91;update&#93;</a></sup> IANA distinguishes the following groups of top-level domains:<sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li><a href="#Infrastructure_domain">Infrastructure top-level domain</a> (ARPA): This group consists of one domain, the <a href="/wiki/.arpa" title=".arpa">Address and Routing Parameter Area</a>. It is managed by IANA on behalf of the <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">Internet Engineering Task Force</a> for various purposes specified in the <a href="/wiki/Request_for_Comments" title="Request for Comments">Request for Comments</a> publications.</li>
<li><a href="/wiki/Generic_top-level_domain" title="Generic top-level domain">Generic top-level domains</a> (gTLD): Top-level domains with three or more characters</li>
<li>Generic restricted top-level domains (grTLD): These domains are managed under official ICANN-accredited registrars.</li>
<li><a href="/wiki/Sponsored_top-level_domain" title="Sponsored top-level domain">Sponsored top-level domains</a> (sTLD): These domains are proposed and sponsored by private agencies or organizations that establish and enforce rules restricting the eligibility to use the TLD. Use is based on community theme concepts; these domains are managed under official ICANN accredited registrars.</li>
<li><a href="/wiki/Country_code_top-level_domain" title="Country code top-level domain">country-code top-level domains</a> (ccTLD): Two-letter domains established for <a href="/wiki/Country" title="Country">countries</a> or <a href="/wiki/List_of_dependent_territories" class="mw-redirect" title="List of dependent territories">territories</a>. With some historical exceptions, the code for any territory is the same as its two-letter <a href="/wiki/ISO_3166" title="ISO 3166">ISO 3166</a> code.
<ul><li><a href="/wiki/Internationalized_country_code_top-level_domain" title="Internationalized country code top-level domain">Internationalized country code top-level domains</a> (IDN ccTLD): ccTLDs in non-Latin character sets (e.g., Arabic, Cyrillic, Greek, Hebrew, or Chinese).</li></ul></li>
<li><a href="/wiki/.test" title=".test">Test top-level domains</a> (tTLD): These domains were installed under <a href="/wiki/.test" title=".test">.test</a> for testing purposes in the IDN development process; these domains are not present in the root zone.</li></ul>
<p>Countries are designated in the Domain Name System by their <a href="/wiki/ISO_3166-1_alpha-2" title="ISO 3166-1 alpha-2">two-letter ISO country code</a>;<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> there are exceptions, however (e.g., <a href="/wiki/.uk" title=".uk">.uk</a>). This group of domains is, therefore, commonly known as <a href="/wiki/Country-code_top-level_domain" class="mw-redirect" title="Country-code top-level domain">country-code top-level domains</a> (ccTLD). Since 2009, countries with non–Latin-based scripts may apply for <a href="/wiki/Internationalized_country_code_top-level_domain" title="Internationalized country code top-level domain">internationalized country code top-level domain</a> names, which are displayed in end-user applications in their language-native script or alphabet, but use a <a href="/wiki/Punycode" title="Punycode">Punycode</a>-translated ASCII domain name in the Domain Name System.
</p><p>Generic top-level domains (formerly <i>categories</i>) initially consisted of <a href="/wiki/.gov" title=".gov">.gov</a>, <a href="/wiki/.edu" title=".edu">.edu</a>, <a href="/wiki/.com" title=".com">.com</a>, <a href="/wiki/.mil" title=".mil">.mil</a>, <a href="/wiki/.org" title=".org">.org</a>, and <a href="/wiki/.net" title=".net">.net</a>. More generic TLDs have been added, such as <a href="/wiki/.info" title=".info">.info</a>.
</p><p>The authoritative <a href="/wiki/List_of_Internet_top-level_domains" title="List of Internet top-level domains">list of current TLDs</a> in the root zone is published at the IANA website at <a rel="nofollow" class="external text" href="https://www.iana.org/domains/root/db/">https://www.iana.org/domains/root/db/</a> as well as <a rel="nofollow" class="external text" href="https://data.iana.org/TLD/tlds-alpha-by-domain.txt">tlds-alpha-by-domain.txt</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="Internationalized_country_code_TLDs">Internationalized country code TLDs</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=3" title="Edit section: Internationalized country code TLDs"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>An <a href="/wiki/Internationalized_country_code_top-level_domain" title="Internationalized country code top-level domain">internationalized country code top-level domain</a> (IDN ccTLD) is a top-level domain with a specially encoded domain name that is displayed in an end-user application, such as a <a href="/wiki/Web_browser" title="Web browser">web browser</a>, in its language-native script or <a href="/wiki/Alphabet" title="Alphabet">alphabet</a> (such as the <a href="/wiki/Arabic_alphabet" title="Arabic alphabet">Arabic alphabet</a>), or a non-alphabetic <a href="/wiki/Writing_system" title="Writing system">writing system</a> (such as <a href="/wiki/Chinese_character" class="mw-redirect" title="Chinese character">Chinese characters</a>). IDN ccTLDs are an application of the <a href="/wiki/Internationalized_domain_name" title="Internationalized domain name">internationalized domain name</a> (IDN) system to top-level Internet domains assigned to countries, or independent geographic regions.
</p><p>ICANN started to accept applications for IDN ccTLDs in November 2009,<sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> and installed the first set into the Domain Names System in May 2010. The first set was a group of Arabic names for the countries of Egypt, Saudi Arabia, and the United Arab Emirates. By May 2010, 21 countries had submitted applications to ICANN, representing 11 scripts.<sup id="cite&#95;ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Infrastructure_domain">Infrastructure domain</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=4" title="Edit section: Infrastructure domain"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The domain <a href="/wiki/.arpa" title=".arpa">.arpa</a> was the first Internet top-level domain. It was intended to be used only temporarily, aiding in the transition of traditional ARPANET host names to the domain name system. However, after it had been used for <a href="/wiki/Reverse_DNS_lookup" title="Reverse DNS lookup">reverse DNS lookup</a>, it was found impractical to retire it, and is used today exclusively for Internet infrastructure purposes such as in-addr.arpa for <a href="/wiki/IPv4" title="IPv4">IPv4</a> and ip6.arpa for IPv6 reverse DNS resolution, uri.arpa and urn.arpa for the <a href="/wiki/Dynamic_Delegation_Discovery_System" title="Dynamic Delegation Discovery System">Dynamic Delegation Discovery System</a>, and e164.arpa for <a href="/wiki/Telephone_number_mapping" title="Telephone number mapping">telephone number mapping</a> based on <a href="/wiki/NAPTR_record" title="NAPTR record">NAPTR DNS records</a>. For historical reasons, <a href="/wiki/.arpa" title=".arpa">.arpa</a> is sometimes considered to be a generic top-level domain.<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Reserved_domains">Reserved domains</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=5" title="Edit section: Reserved domains"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>A set of domain names is reserved<sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup> by the Internet Engineering Task Force as <a href="/wiki/Special-use_domain_name" title="Special-use domain name">special-use domain names</a>. The practice originated in RFC 1597 for reserved address allocations in 1994 and reserved top-level domains in RFC 2606 of 1999, with additional reservations in later RFCs. These reserved names should not be used in production networks that utilize the global domain name system.
</p>

<p>In the late 1980s, <a href="/wiki/InterNIC" title="InterNIC">InterNIC</a> created the <a href="/wiki/.nato" title=".nato">.nato</a> domain for use by <a href="/wiki/NATO" title="NATO">NATO</a>.<sup id="cite&#95;ref-17" class="reference"><a href="#cite_note-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup> NATO considered none of the then-existing TLDs as adequately reflecting their status as an <a href="/wiki/International_organization" title="International organization">international organization</a>. Soon after this addition, however, InterNIC also created the <a href="/wiki/.int" title=".int">.int</a> TLD for the use by international organizations in general, and persuaded NATO to use the second level domain <i>nato.int</i> instead. The <i>nato</i> TLD, no longer used, was finally removed in July 1996.<sup id="cite&#95;ref-18" class="reference"><a href="#cite_note-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Other historical TLDs are <a href="/wiki/.cs" title=".cs">.cs</a> for <a href="/wiki/Czechoslovakia" title="Czechoslovakia">Czechoslovakia</a> (now using <a href="/wiki/.cz" title=".cz">.cz</a> for <a href="/wiki/Czech_Republic" title="Czech Republic">Czech Republic</a> and <a href="/wiki/.sk" title=".sk">.sk</a> for <a href="/wiki/Slovakia" title="Slovakia">Slovakia</a>), <a href="/wiki/.dd" title=".dd">.dd</a> for <a href="/wiki/East_Germany" title="East Germany">East Germany</a> (using <a href="/wiki/.de" title=".de">.de</a> after <a href="/wiki/Reunification_of_Germany" class="mw-redirect" title="Reunification of Germany">reunification of Germany</a>), <a href="/wiki/.yu" title=".yu">.yu</a> for <a href="/wiki/Socialist_Federal_Republic_of_Yugoslavia" title="Socialist Federal Republic of Yugoslavia">SFR Yugoslavia</a> and <a href="/wiki/Serbia_and_Montenegro" title="Serbia and Montenegro">Serbia and Montenegro</a> (now using <a href="/wiki/.ba" title=".ba">.ba</a> for <a href="/wiki/Bosnia_and_Herzegovina" title="Bosnia and Herzegovina">Bosnia and Herzegovina</a>, <a href="/wiki/.hr" title=".hr">.hr</a> for <a href="/wiki/Croatia" title="Croatia">Croatia</a>, <a href="/wiki/.me" title=".me">.me</a> for <a href="/wiki/Montenegro" title="Montenegro">Montenegro</a>, <a href="/wiki/.mk" title=".mk">.mk</a> for <a href="/wiki/North_Macedonia" title="North Macedonia">North Macedonia</a>, <a href="/wiki/.rs" title=".rs">.rs</a> for <a href="/wiki/Serbia" title="Serbia">Serbia</a> and <a href="/wiki/.si" title=".si">.si</a> for <a href="/wiki/Slovenia" title="Slovenia">Slovenia</a>), <a href="/wiki/.zr" class="mw-redirect" title=".zr">.zr</a> for <a href="/wiki/Zaire" title="Zaire">Zaire</a> (now <a href="/wiki/.cd" title=".cd">.cd</a> for the <a href="/wiki/Democratic_Republic_of_the_Congo" title="Democratic Republic of the Congo">Democratic Republic of the Congo</a>), and <a href="/wiki/.an" title=".an">.an</a> for <a href="/wiki/Netherlands_Antilles" title="Netherlands Antilles">Netherlands Antilles</a> (now <a href="/wiki/.aw" title=".aw">.aw</a> for <a href="/wiki/Aruba" title="Aruba">Aruba</a>, <a href="/wiki/.cw" title=".cw">.cw</a> for <a href="/wiki/Cura%C3%A7ao" title="Curaçao">Curaçao</a> and <a href="/wiki/.sx" title=".sx">.sx</a> for <a href="/wiki/Sint_Maarten" title="Sint Maarten">Sint Maarten</a>). In contrast to these, the TLD <a href="/wiki/.su" title=".su">.su</a> has remained active despite the collapse of the <a href="/wiki/Soviet_Union" title="Soviet Union">Soviet Union</a> that it represents. Under the chairmanship of <a href="/wiki/Nigel_Roberts" title="Nigel Roberts">Nigel Roberts</a>, <a href="/wiki/ICANN" title="ICANN">ICANN</a>'s <a href="/wiki/CcNSO" class="mw-redirect" title="CcNSO">ccNSO</a> is working on a policy for the retirement of <a href="/wiki/CcTLD" class="mw-redirect" title="CcTLD">ccTLDs</a> that have been removed from <a href="/wiki/ISO_3166" title="ISO 3166">ISO 3166</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="Proposed_domains">Proposed domains</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=7" title="Edit section: Proposed domains"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Further information: <a href="/wiki/Proposed_top-level_domain" title="Proposed top-level domain">Proposed top-level domain</a></div>
<p>Around late 2000, <a href="/wiki/ICANN" title="ICANN">ICANN</a> discussed and finally introduced<sup id="cite&#95;ref-19" class="reference"><a href="#cite_note-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/.aero" title=".aero">.aero</a>, <a href="/wiki/.biz" title=".biz">.biz</a>, <a href="/wiki/.coop" title=".coop">.coop</a>, <a href="/wiki/.info" title=".info">.info</a>, <a href="/wiki/.museum" title=".museum">.museum</a>, <a href="/wiki/.name" title=".name">.name</a>, and <a href="/wiki/.pro" title=".pro">.pro</a> TLDs. Site owners argued that a similar TLD should be made available for adult and pornographic websites to settle the dispute of obscene content on the Internet, to address the responsibility of US service providers under the US <a href="/wiki/Communications_Decency_Act" title="Communications Decency Act">Communications Decency Act</a> of 1996. Several options were proposed including <i>xxx</i>, <i>sex</i> and <i>adult</i>.<sup id="cite&#95;ref-20" class="reference"><a href="#cite_note-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup> The <a href="/wiki/.xxx" title=".xxx">.xxx</a> top-level domain eventually went live in 2011.<sup id="cite&#95;ref-21" class="reference"><a href="#cite_note-21"><span class="cite-bracket">&#91;</span>21<span class="cite-bracket">&#93;</span></a></sup>
</p><p>An older proposal consisted of seven new gTLDs: arts, firm, <a href="/wiki/.info" title=".info">.info</a>, nom, rec, <a href="/wiki/.shop" title=".shop">.shop</a>, and <a href="/wiki/.web" title=".web">.web</a>.<sup id="cite&#95;ref-22" class="reference"><a href="#cite_note-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup> Later <a href="/wiki/.biz" title=".biz">.biz</a>, <a href="/wiki/.info" title=".info">.info</a>, <a href="/wiki/.museum" title=".museum">.museum</a>, and <a href="/wiki/.name" title=".name">.name</a> covered most of these old proposals.
</p><p>During the 32nd International Public ICANN Meeting in Paris in 2008, ICANN started a new process of TLD naming policy to take a "significant step forward on the introduction of new generic top-level domains".<sup id="cite&#95;ref-23" class="reference"><a href="#cite_note-23"><span class="cite-bracket">&#91;</span>23<span class="cite-bracket">&#93;</span></a></sup> This program envisioned the availability of many new or already proposed domains, as well as a new application and implementation process.<sup id="cite&#95;ref-24" class="reference"><a href="#cite_note-24"><span class="cite-bracket">&#91;</span>24<span class="cite-bracket">&#93;</span></a></sup> Observers believed that the new rules could result in hundreds of new gTLDs being registered.<sup id="cite&#95;ref-25" class="reference"><a href="#cite_note-25"><span class="cite-bracket">&#91;</span>25<span class="cite-bracket">&#93;</span></a></sup>
</p><p>On 13 June 2012, <a href="/wiki/ICANN" title="ICANN">ICANN</a> announced nearly 2,000 applications for top-level domains, which began installation throughout 2013.<sup id="cite&#95;ref-26" class="reference"><a href="#cite_note-26"><span class="cite-bracket">&#91;</span>26<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-27" class="reference"><a href="#cite_note-27"><span class="cite-bracket">&#91;</span>27<span class="cite-bracket">&#93;</span></a></sup> The first seven – <i>bike</i>, <i>clothing</i>, <i>guru</i>, <i>holdings</i>, <i>plumbing</i>, <i>singles</i>, and <i>ventures</i> – were released in 2014.<sup id="cite&#95;ref-28" class="reference"><a href="#cite_note-28"><span class="cite-bracket">&#91;</span>28<span class="cite-bracket">&#93;</span></a></sup> As of 2025, there are approximately 1,200 delegated generic top-level domains (gTLDs) in the global Domain Name System, including legacy gTLDs (such as .com, .org, .net) and gTLDs introduced in the 2012 New gTLD Program<sup id="cite&#95;ref-29" class="reference"><a href="#cite_note-29"><span class="cite-bracket">&#91;</span>29<span class="cite-bracket">&#93;</span></a></sup>. 
</p><p><a href="/wiki/ICANN" title="ICANN">ICANN</a> has been gearing to roll out the subsequent round of New gTLD expansion but opening the second round of New gTLD Application window in April 2026 for 12 - 15 weeks. The first draft of New gTLD Applicant Guidebook was published on 30 May 2025 followed by the period of public comments has now been approved by the ICANN board and is now expected to be published in ICANN's website by December 2025.
</p><p>ICANN has also opened up Applicant Support Program<sup id="cite&#95;ref-30" class="reference"><a href="#cite_note-30"><span class="cite-bracket">&#91;</span>30<span class="cite-bracket">&#93;</span></a></sup> to support organizations and communities which are in need of financial support to apply for New gTLD. The application window to apply for financial support has been extended until 19th December 2025 from the original deadline of 19th November 2025. ICANN has also mandated applicants to choose a registry backend service provider from ICANN's accredited Registry Service Provider<sup id="cite&#95;ref-31" class="reference"><a href="#cite_note-31"><span class="cite-bracket">&#91;</span>31<span class="cite-bracket">&#93;</span></a></sup> (RSP) list which is expected to be published in early 2026. ICANN is publishing timely updates on their website. &#160;
</p>
<div class="mw-heading mw-heading3"><h3 id="Rejected_domains">Rejected domains</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=8" title="Edit section: Rejected domains"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>ICANN rejected several proposed domains to include <a href="/wiki/.home" title=".home">.home</a> and <a href="/wiki/.corp" title=".corp">.corp</a> due to conflicts regarding gTLDs that are in use in internal networks.
</p><p>Investigation into the conflicts was conducted at ICANN's request by <a href="/w/index.php?title=Interisle_Consulting&amp;action=edit&amp;redlink=1" class="new" title="Interisle Consulting (page does not exist)">Interisle Consulting</a>. The resulting report was to become known as the Name Collision<sup id="cite&#95;ref-32" class="reference"><a href="#cite_note-32"><span class="cite-bracket">&#91;</span>32<span class="cite-bracket">&#93;</span></a></sup> issue, which was first reported at ICANN 47.<sup id="cite&#95;ref-33" class="reference"><a href="#cite_note-33"><span class="cite-bracket">&#91;</span>33<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Dotless_domains">Dotless domains</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=9" title="Edit section: Dotless domains"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:DNS_schema.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/d/d2/DNS_schema.svg/250px-DNS_schema.svg.png" decoding="async" width="250" height="139" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/d2/DNS_schema.svg/500px-DNS_schema.svg.png 1.5x" data-file-width="1073" data-file-height="596" /></a><figcaption><i><a href="/wiki/.org" title=".org">.org</a>[.]</i> is a node in the DNS tree, just like <i>wikipedia.[org.]</i> and <i>en.[wikipedia.org.]</i>. As such, it has its own <a href="/wiki/DNS_record" class="mw-redirect" title="DNS record">DNS records</a>.</figcaption></figure>
<p>Due to the structure of <a href="/wiki/DNS" class="mw-redirect" title="DNS">DNS</a>, each node in the tree has its own collection of <a href="/wiki/DNS_record" class="mw-redirect" title="DNS record">records</a>, and since top-level domains are nodes in DNS, they have records of their own. For example, querying <i>org</i> itself (with a tool such as <a href="/wiki/Dig_(command)" title="Dig (command)">dig</a>, <a href="/wiki/Host_(Unix)" title="Host (Unix)">host</a>, or <a href="/wiki/Nslookup" title="Nslookup">nslookup</a>) returns information on its <a href="/wiki/Nameserver" class="mw-redirect" title="Nameserver">nameservers</a>:
</p>
<div style="font-size:85%" class="mw-highlight mw-highlight-lang-zone mw-content-ltr" dir="ltr"><pre><span></span><span class="hll"><span class="err">QUESTION</span>
</span><span class="n">org.</span><span class="w"> </span><span class="kt">IN</span><span class="w"> </span><span class="l">ANY</span>
<span class="hll"><span class="err">ANSWER</span>

<p><i>Dotless domains</i> are top-level domains that take advantage of that fact, and implement <a href="/wiki/A_record" class="mw-redirect" title="A record">A</a>, <a href="/wiki/AAAA_record" class="mw-redirect" title="AAAA record">AAAA</a> or <a href="/wiki/MX_record" title="MX record">MX</a> DNS records to serve webpages or allow incoming email directly on a TLD – for example, a webpage hosted on <i>http://example/</i>, or an email address <i>user@example</i>.<sup id="cite&#95;ref-dotless-icann&#95;34-0" class="reference"><a href="#cite_note-dotless-icann-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup>
</p><p><a href="/wiki/ICANN" title="ICANN">ICANN</a> and <a href="/wiki/Internet_Architecture_Board" title="Internet Architecture Board">IAB</a> have spoken out against the practice, classifying it as a security risk among other concerns.<sup id="cite&#95;ref-35" class="reference"><a href="#cite_note-35"><span class="cite-bracket">&#91;</span>35<span class="cite-bracket">&#93;</span></a></sup> ICANN's Security and Stability Advisory Committee (SSAC) additionally claims that <a href="/wiki/SMTP" class="mw-redirect" title="SMTP">SMTP</a> "requires at least two labels in the <a href="/wiki/Fully_qualified_domain_name" title="Fully qualified domain name">FQDN</a> of a mail address" and, as such, mail servers would reject emails to addresses with dotless domains.<sup id="cite&#95;ref-dotless-icann&#95;34-1" class="reference"><a href="#cite_note-dotless-icann-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup>
</p><p>ICANN has also published a resolution in 2013 that prohibits the creation of dotless domains on <a href="/wiki/GTLD" class="mw-redirect" title="GTLD">gTLDs</a>.<sup id="cite&#95;ref-36" class="reference"><a href="#cite_note-36"><span class="cite-bracket">&#91;</span>36<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/CcTLD" class="mw-redirect" title="CcTLD">ccTLDs</a>, however, fall largely under their respective country's <a href="/wiki/Jurisdiction" title="Jurisdiction">jurisdiction</a>, and not under ICANN's. Because of this, there have been many examples of dotless domains on ccTLDs in spite of ICANN's vocal opposition.
</p><p>As of July 2025, that is the case of:
</p>
<ul><li><a href="/wiki/Uzbekistan" title="Uzbekistan">Uzbekistan</a>'s <a href="/wiki/.uz" title=".uz">.uz</a>, online at <a rel="nofollow" class="external free" href="https://uz./">https://uz./</a> (<a href="/wiki/Wikipedia:Archive.today_guidance" title="Wikipedia:Archive.today guidance">Deprecated link</a> archived 4 September 2023 at <a href="/wiki/Archive.today" title="Archive.today">archive.today</a>)<br />It's a mirror of <a rel="nofollow" class="external free" href="https://cctld.uz/">https://cctld.uz/</a>, albeit with an invalid certificate.</li></ul>
<p>In 2023, it used to be the case of:
</p>
<ul><li><a href="/wiki/Anguilla" title="Anguilla">Anguilla</a>'s <a href="/wiki/.ai" title=".ai">.ai</a>, online at <a rel="nofollow" class="external free" href="http://ai./">http://ai./</a> (<a href="/wiki/Wikipedia:Archive.today_guidance" title="Wikipedia:Archive.today guidance">Deprecated link</a> archived 22 January 2023 at <a href="/wiki/Archive.today" title="Archive.today">archive.today</a>)<br />It simply displayed a notice that the website was no longer public.<br />The TLD no longer has an A or AAAA record.</li></ul>
<p>Other ccTLDs with A or AAAA records, as of July 2025, include: <a href="/wiki/.cm" title=".cm">.cm</a>, <a href="/wiki/.tk" title=".tk">.tk</a> and <a href="/wiki/.ws" title=".ws">.ws</a>.
</p><p>A similar query to <i>org</i><span class="nowrap" style="padding-left:0.1em;">&#39;</span>s presented above could be made for <i>ai</i>, which showed A and MX records for the TLD:
</p>

<p>Historically, many other ccTLDs have had A or AAAA records. On 3 September 2013, as reported by the <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a>, they were the following:<sup id="cite&#95;ref-37" class="reference"><a href="#cite_note-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/.ac" title=".ac">.ac</a>, <a href="/wiki/.dk" title=".dk">.dk</a>, <a href="/wiki/.gg" title=".gg">.gg</a>, <a href="/wiki/.io" title=".io">.io</a>, <a href="/wiki/.je" title=".je">.je</a>, <a href="/wiki/.kh" title=".kh">.kh</a>, <a href="/wiki/.sh" title=".sh">.sh</a>, <a href="/wiki/.tm" title=".tm">.tm</a>, <a href="/wiki/.to" title=".to">.to</a>, and <a href="/wiki/.vi" title=".vi">.vi</a>.
</p>
<div class="mw-heading mw-heading3"><h3 id="New_TLDs">New TLDs</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=10" title="Edit section: New TLDs"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Following a 2014 resolution by ICANN, newly registered TLDs must implement the following A, MX, TXT, and SRV apex DNS records – where <code>&lt;TLD&gt;</code> stands for the registered TLD – for at least 90 days:<sup id="cite&#95;ref-38" class="reference"><a href="#cite_note-38"><span class="cite-bracket">&#91;</span>38<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div style="font-size:85%" class="mw-highlight mw-highlight-lang-zone mw-content-ltr" dir="ltr"><pre><span></span><span class="n">&lt;TLD&gt;.</span><span class="w"> </span><span class="mi">3600</span><span class="w"> </span><span class="nc">IN</span><span class="w"> </span><span class="kt">MX</span><span class="w">  </span><span class="mi">10</span><span class="w">      </span><span class="l">your-dns-needs-immediate-attention.&lt;TLD&gt;.</span>
<span class="n">&lt;TLD&gt;.</span><span class="w"> </span><span class="mi">3600</span><span class="w"> </span><span class="nc">IN</span><span class="w"> </span><span class="kt">SRV</span><span class="w"> </span><span class="mi">10</span><span class="w"> </span><span class="mi">10</span><span class="w"> </span><span class="mi">0</span><span class="w"> </span><span class="l">your-dns-needs-immediate-attention.&lt;TLD&gt;.</span>
<span class="n">&lt;TLD&gt;.</span><span class="w"> </span><span class="mi">3600</span><span class="w"> </span><span class="nc">IN</span><span class="w"> </span><span class="kt">TXT</span><span class="w">         </span><span class="s">&quot;Your DNS configuration needs immediate attention see https://icann.org/namecollision&quot;</span>
<span class="n">&lt;TLD&gt;.</span><span class="w"> </span><span class="mi">3600</span><span class="w"> </span><span class="nc">IN</span><span class="w"> </span><span class="kt">A</span><span class="w">           </span><span class="l">127.0.53.53</span>
</pre></div>
<p>This requirement is meant to avoid domain name collisions when new TLDs are registered. For example, programmers may have used custom local domains such as <i>foo.bar</i> or <i>test.dev</i>, which would both collide with the creation of gTLDs <a href="/wiki/.bar" title=".bar">.bar</a> in 2014 and <a href="/wiki/.dev" title=".dev">.dev</a> in 2019.
</p><p>While this does create apex DNS records of type A and MX, they do not qualify as a dotless domain, as the records should not point to real servers. For instance, the A record contains the IP 127.0.53.53, a loopback address (see <a href="/wiki/IPv4#Addressing" title="IPv4">IPv4 § Addressing</a>), picked as a mnemonic to indicate a DNS-related problem, as <a href="/wiki/List_of_TCP_and_UDP_port_numbers" title="List of TCP and UDP port numbers">DNS uses port 53</a>.<sup id="cite&#95;ref-39" class="reference"><a href="#cite_note-39"><span class="cite-bracket">&#91;</span>39<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Pseudo-domains">Pseudo-domains</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Top-level_domain&amp;action=edit&amp;section=11" title="Edit section: Pseudo-domains"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Pseudo-top-level_domain" title="Pseudo-top-level domain">Pseudo-top-level domain</a></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1305433154" /><table class="box-More&#95;citations&#95;needed plainlinks metadata ambox ambox-content ambox-Refimprove" role="presentation"><tbody><tr><td class="mbox-image"><div class="mbox-image-div"><span typeof="mw:File"><a href="/wiki/File:Question_book-new.svg" class="mw-file-description"><img alt="icon" src="//upload.wikimedia.org/wikipedia/en/thumb/9/99/Question_book-new.svg/60px-Question_book-new.svg.png" decoding="async" width="50" height="39" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/en/thumb/9/99/Question_book-new.svg/120px-Question_book-new.svg.png 1.5x" data-file-width="512" data-file-height="399" /></a></span></div></td><td class="mbox-text"><div class="mbox-text-span">This section <b>needs additional citations for <a href="/wiki/Wikipedia:Verifiability" title="Wikipedia:Verifiability">verification</a></b>.<span class="hide-when-compact"> Please help <a href="/wiki/Special:EditPage/Top-level_domain" title="Special:EditPage/Top-level domain">improve this article</a> by <a href="/wiki/Help:Referencing_for_beginners" title="Help:Referencing for beginners">adding citations to reliable sources</a>&#32;in this section. Unsourced material may be challenged and removed.<br /><small><span class="plainlinks"><i>Find sources:</i>&#160;<a rel="nofollow" class="external text" href="https://www.google.com/search?as_eq=wikipedia&amp;q=%22Top-level+domain%22">"Top-level domain"</a>&#160;–&#160;<a rel="nofollow" class="external text" href="https://www.google.com/search?tbm=nws&amp;q=%22Top-level+domain%22+-wikipedia&amp;tbs=ar:1">news</a>&#160;<b>·</b> <a rel="nofollow" class="external text" href="https://www.google.com/search?&amp;q=%22Top-level+domain%22&amp;tbs=bkt:s&amp;tbm=bks">newspapers</a>&#160;<b>·</b> <a rel="nofollow" class="external text" href="https://www.google.com/search?tbs=bks:1&amp;q=%22Top-level+domain%22+-wikipedia">books</a>&#160;<b>·</b> <a rel="nofollow" class="external text" href="https://scholar.google.com/scholar?q=%22Top-level+domain%22">scholar</a>&#160;<b>·</b> <a rel="nofollow" class="external text" href="https://www.jstor.org/action/doBasicSearch?Query=%22Top-level+domain%22&amp;acc=on&amp;wc=on">JSTOR</a></span></small></span>  <span class="date-container"><i>(<span class="date">December 2018</span>)</i></span><span class="hide-when-compact"><i> (<small><a href="/wiki/Help:Maintenance_template_removal" title="Help:Maintenance template removal">Learn how and when to remove this message</a></small>)</i></span></div></td></tr></tbody></table>
<p>Several networks, such as <a href="/wiki/BITNET" title="BITNET">BITNET</a>, <a href="/wiki/CSNET" title="CSNET">CSNET</a>, and <a href="/wiki/UUCP" title="UUCP">UUCP</a>, existed that were in widespread use among computer professionals and academic users, but were not interoperable directly with the Internet and exchanged mail with the Internet via special email gateways. For relaying purposes on the gateways, messages associated with these networks were labeled with suffixes such as <a href="/wiki/.bitnet" class="mw-redirect" title=".bitnet">.bitnet</a>, <a href="/wiki/.oz" class="mw-redirect" title=".oz">.oz</a>, <a href="/wiki/.csnet" class="mw-redirect" title=".csnet">.csnet</a>, or <a href="/wiki/.uucp" class="mw-redirect" title=".uucp">.uucp</a>, but these domains did not exist as top-level domains in the public <a href="/wiki/Domain_Name_System" title="Domain Name System">Domain Name System</a> of the Internet.
</p><p>Most of these networks have long since ceased to exist, and although UUCP still gets significant use in parts of the world where Internet infrastructure has not yet become well established, it subsequently transitioned to using Internet domain names, and pseudo-domains now largely survive as historical relics. One notable exception is the 2007 emergence of <a href="/wiki/Society_for_Worldwide_Interbank_Financial_Telecommunication" class="mw-redirect" title="Society for Worldwide Interbank Financial Telecommunication">SWIFTNet</a> Mail, which uses the swift pseudo-domain.<sup id="cite&#95;ref-40" class="reference"><a href="#cite_note-40"><span class="cite-bracket">&#91;</span>40<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The anonymity network <a href="/wiki/Tor_(network)" title="Tor (network)">Tor</a> formerly used the top-level pseudo-domain <a href="/wiki/.onion" title=".onion">.onion</a> for <a href="/wiki/Onion_services" class="mw-redirect" title="Onion services">onion services</a>, which can only be reached with a Tor client because it uses the Tor <a href="/wiki/Onion_routing" title="Onion routing">onion routing</a> protocol to reach the hidden service to protect the anonymity of users. However, the pseudo-domain became officially reserved in October 2015.<sup id="cite&#95;ref-41" class="reference"><a href="#cite_note-41"><span class="cite-bracket">&#91;</span>41<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/I2p" class="mw-redirect" title="I2p">i2p</a> provides a similar hidden pseudo-domain, .i2p,<sup id="cite&#95;ref-42" class="reference"><a href="#cite_note-42"><span class="cite-bracket">&#91;</span>42<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/Namecoin" title="Namecoin">Namecoin</a> uses the .bit pseudo-domain.<sup id="cite&#95;ref-43" class="reference"><a href="#cite_note-43"><span class="cite-bracket">&#91;</span>43<span class="cite-bracket">&#93;</span></a></sup>
</p>
