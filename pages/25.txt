					<div id="contentSub"><div id="mw-content-subtitle"><span class="mw-redirectedfrom">(Redirected from <a href="/w/index.php?title=URL_normalization&amp;redirect=no" class="mw-redirect" title="URL normalization">URL normalization</a>)</span></div></div>
					
					
					<div id="mw-content-text" class="mw-body-content"><div class="mw-subjectpageheader">
</div><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">Process by which URIs are standardized</div>
<p class="mw-empty-elt">

</p>
<style data-mw-deduplicate="TemplateStyles:r1320445320">.mw-parser-output .hatnote{font-style:italic}.mw-parser-output div.hatnote{padding-left:1.6em;margin-bottom:0.5em}.mw-parser-output .hatnote i{font-style:normal}.mw-parser-output .hatnote+link+.hatnote{margin-top:-0.5em}@media print{body.ns-0 .mw-parser-output .hatnote{display:none!important}}</style><div role="note" class="hatnote navigation-not-searchable">Not to be confused with <a href="/wiki/URL_canonicalization" class="mw-redirect" title="URL canonicalization">URL canonicalization</a>.</div>
<figure class="mw-halign-right" typeof="mw:File/Thumb"><a href="/wiki/File:Normalization_URL_Animation.gif" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/6/64/Normalization_URL_Animation.gif" decoding="async" width="300" height="300" class="mw-file-element" data-file-width="300" data-file-height="300" /></a><figcaption>Types of URI normalization.</figcaption></figure>
<p><b>URI normalization</b> is the process by which <a href="/wiki/Uniform_Resource_Identifier" title="Uniform Resource Identifier">URIs</a> are modified and standardized in a consistent manner. The goal of the normalization process is to transform a URI into a normalized URI so it is possible to determine if two syntactically different URIs may be equivalent.
</p><p><a href="/wiki/Search_engine" title="Search engine">Search engines</a> employ URI normalization in order to correctly rank pages that may be found with multiple URIs, and to reduce indexing of duplicate pages. <a href="/wiki/Web_crawler" title="Web crawler">Web crawlers</a> perform URI normalization in order to avoid crawling the same resource more than once. <a href="/wiki/Web_browser" title="Web browser">Web browsers</a> may perform normalization to determine if a link has been visited or to determine if a <a href="/wiki/Web_cache" title="Web cache">page has been cached</a>. <a href="/wiki/Web_server" title="Web server">Web servers</a> may also perform normalization for many reasons (i.e. to be able to more easily intercept security risks coming from client requests, to use only one absolute file name for each resource stored in their caches, named in log files, etc.).
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Normalization_process">Normalization process</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=URI_normalization&amp;action=edit&amp;section=1" title="Edit section: Normalization process"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>There are several types of normalization that may be performed. Some of them are always semantics preserving and some may not be.
</p>
<div class="mw-heading mw-heading3"><h3 id="Normalizations_that_preserve_semantics">Normalizations that preserve semantics</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=URI_normalization&amp;action=edit&amp;section=2" title="Edit section: Normalizations that preserve semantics"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The following normalizations are described in RFC 3986 <sup id="cite&#95;ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> to result in equivalent URIs:
</p>
<ul><li><b>Converting percent-encoded triplets to uppercase.</b> The <a href="/wiki/Hexadecimal" title="Hexadecimal">hexadecimal</a> digits within a <a href="/wiki/Percent-encoding" title="Percent-encoding">percent-encoding</a> triplet of the URI (e.g., <code>%3a</code> versus <code>%3A</code>) are <a href="/wiki/Case_sensitivity" title="Case sensitivity">case-insensitive</a> and therefore should be normalized to use uppercase letters for the digits A-F.<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> Example:</li></ul>
<dl><dd><code>http://example.com/foo%2a</code> → <code>http://example.com/foo%2A</code></dd></dl>
<ul><li><b>Converting the scheme and host to lowercase.</b> The <a href="/wiki/URI_scheme" class="mw-redirect" title="URI scheme">scheme</a> and <a href="/wiki/Host_(network)" title="Host (network)">host</a> components of the URI are case-insensitive and therefore should be normalized to lowercase.<sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> Example:</li></ul>
<dl><dd><code>HTTP://User@Example.COM/Foo</code> → <code>http://User@example.com/Foo</code></dd></dl>
<ul><li><b>Decoding percent-encoded triplets of unreserved characters.</b> Percent-encoded triplets of the URI in the ranges of <i>ALPHA</i> (<code>%41</code>–<code>%5A</code> and <code>%61</code>–<code>%7A</code>), <i>DIGIT</i> (<code>%30</code>–<code>%39</code>), hyphen (<code>%2D</code>), period (<code>%2E</code>), underscore (<code>%5F</code>), or tilde (<code>%7E</code>) do not require percent-encoding and should be decoded to their corresponding unreserved characters.<sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> Example:</li></ul>
<dl><dd><code>http://example.com/%7Efoo</code> → <code>http://example.com/~foo</code></dd></dl>
<ul><li><b>Removing dot-segments.</b> Dot-segments <code>.</code> and <code>..</code> in the path component of the URI should be removed by applying the remove_dot_segments algorithm<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> to the path described in RFC 3986.<sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> Example:</li></ul>
<dl><dd><code>http://example.com/foo/./bar/baz/../qux</code> → <code>http://example.com/foo/bar/qux</code></dd></dl>
<ul><li><b>Converting an empty path to a "/" path.</b> In presence of an authority component, an empty path component should be normalized to a path component of "/".<sup id="cite&#95;ref-:0&#95;7-0" class="reference"><a href="#cite_note-:0-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> Example:</li></ul>
<dl><dd><code>http://example.com</code> → <code>http://example.com/</code></dd></dl>
<ul><li><b>Removing the default port.</b> An empty or <a href="/wiki/List_of_TCP_and_UDP_port_numbers" title="List of TCP and UDP port numbers">default port</a> component of the URI (port 80 for the <code>http</code> scheme) with its ":" <a href="/wiki/Delimiter" title="Delimiter">delimiter</a> should be removed.<sup id="cite&#95;ref-:0&#95;7-1" class="reference"><a href="#cite_note-:0-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> Example:</li></ul>
<dl><dd><code>http://example.com:80/</code> → <code>http://example.com/</code></dd></dl>
<div class="mw-heading mw-heading3"><h3 id="Normalizations_that_usually_preserve_semantics">Normalizations that usually preserve semantics</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=URI_normalization&amp;action=edit&amp;section=3" title="Edit section: Normalizations that usually preserve semantics"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>For http and https URIs, the following normalizations listed in RFC 3986 may result in equivalent URIs, but are not guaranteed to by the standards:
</p>
<ul><li><b>Adding a trailing "/" to a non-empty path.</b> Directories (folders) are indicated with a trailing slash and should be included in URIs. Example:</li></ul>
<dl><dd><code>http://example.com/foo</code> → <code>http://example.com/foo/</code></dd>
<dd>However, there is no way to know if a URI path component represents a directory or not. RFC 3986 notes that if the former URI redirects to the latter URI, then that is an indication that they are equivalent.</dd></dl>
<div class="mw-heading mw-heading3"><h3 id="Normalizations_that_change_semantics">Normalizations that change semantics</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=URI_normalization&amp;action=edit&amp;section=4" title="Edit section: Normalizations that change semantics"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Applying the following normalizations result in a semantically different URI although it may refer to the same resource:
</p>
<ul><li><b>Removing directory index.</b> Default <a href="/wiki/Webserver_directory_index" class="mw-redirect" title="Webserver directory index">directory indexes</a> are generally not needed in URIs. Examples:</li></ul>
<dl><dd><code>http://example.com/a/index.html</code> → <code>http://example.com/a/</code></dd>
<dd><code>http://example.com/default.asp</code> → <code>http://example.com/</code></dd></dl>
<ul><li><b>Removing the fragment.</b> The <a href="/wiki/Fragment_identifier" class="mw-redirect" title="Fragment identifier">fragment</a> component of a URI is never seen by the server and can sometimes be removed. Example:</li></ul>
<dl><dd><code>http://example.com/bar.html#section1</code> → <code>http://example.com/bar.html</code></dd>
<dd>However, <a href="/wiki/Ajax_(programming)" title="Ajax (programming)">AJAX</a> applications frequently use the value in the fragment.</dd></dl>
<ul><li><b>Replacing IP with domain name.</b> Check if the <a href="/wiki/IP_address" title="IP address">IP address</a> maps to a domain name. Example:</li></ul>
<dl><dd><code>http://208.77.188.166/</code> → <code>http://example.com/</code></dd>
<dd>The reverse replacement is rarely safe due to <a href="/wiki/Shared_web_hosting_service" title="Shared web hosting service">virtual web servers</a>.</dd></dl>
<ul><li><b>Limiting protocols.</b> Limiting different <a href="/wiki/Application_layer" title="Application layer">application layer</a> protocols. For example, the “https” scheme could be replaced with “http”. Example:</li></ul>
<dl><dd><code>https://example.com/</code> → <code>http://example.com/</code></dd></dl>
<ul><li><b>Removing duplicate slashes</b> Paths which include two adjacent slashes could be converted to one. Example:</li></ul>
<dl><dd><code>http://example.com/foo//bar.html</code> → <code>http://example.com/foo/bar.html</code></dd></dl>
<ul><li><b>Removing or adding “www” as the first domain label.</b> Some websites operate identically in two Internet domains: one whose least significant label is “www” and another whose name is the result of omitting the least significant label from the name of the first, the latter being known as a <a href="/wiki/Naked_domain" class="mw-redirect" title="Naked domain">naked domain</a>. For example, <code>http://www.example.com/</code> and <code>http://example.com/</code> may access the same website. Many websites <a href="/wiki/URL_redirection" title="URL redirection">redirect</a> the user from the <a href="/wiki/WWW_prefix" class="mw-redirect" title="WWW prefix">www</a> to the non-www address or vice versa. A normalizer may determine if one of these URIs redirects to the other and normalize all URIs appropriately. Example:</li></ul>
<dl><dd><code>http://www.example.com/</code> → <code>http://example.com/</code></dd></dl>
<ul><li><b>Sorting the query parameters.</b> Some web pages use more than one <a href="/wiki/Query_string#Web_forms" title="Query string">query parameter</a> in the URI. A normalizer can sort the parameters into alphabetical order (with their values), and reassemble the URI. Example:</li></ul>
<dl><dd><code>http://example.com/display?lang=en&amp;article=fred</code> → <code>http://example.com/display?article=fred&amp;lang=en</code></dd>
<dd>However, the order of parameters in a URI may be significant (this is not defined by the standard) and a web server may allow the same variable to appear multiple times.<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup></dd></dl>
<ul><li><b>Removing unused query variables.</b> A page may only expect certain parameters to appear in the query; unused parameters can be removed. Example:</li></ul>
<dl><dd><code>http://example.com/display?id=123&amp;fakefoo=fakebar</code> → <code>http://example.com/display?id=123</code></dd>
<dd>Note that a parameter without a value is not necessarily an unused parameter.</dd></dl>
<ul><li><b>Removing default query parameters.</b> A default value in the query string may render identically whether it is there or not. Example:</li></ul>
<dl><dd><code>http://example.com/display?id=&amp;sort=ascending</code> → <code>http://example.com/display</code></dd></dl>
<ul><li><b>Removing the "?" when the query is empty.</b> When the query is empty, there may be no need for the "?". Example:</li></ul>
<dl><dd><code>http://example.com/display?</code> → <code>http://example.com/display</code></dd></dl>
<div class="mw-heading mw-heading2"><h2 id="Normalization_based_on_URI_lists">Normalization based on URI lists</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=URI_normalization&amp;action=edit&amp;section=5" title="Edit section: Normalization based on URI lists"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Some normalization rules may be developed for specific websites by examining URI lists obtained from previous crawls or web server logs. For example, if the URI
</p>
<dl><dd><code>http://example.com/story?id=xyz</code></dd></dl>
<p>appears in a crawl log several times along with
</p>
<dl><dd><code>http://example.com/story_xyz</code></dd></dl>
<p>we may assume that the two URIs are equivalent and can be normalized to one of the URI forms.
</p><p>Schonfeld et al. (2006) present a <a href="/wiki/Heuristic" title="Heuristic">heuristic</a> called DustBuster for detecting DUST (different URIs with similar text) rules that can be applied to URI lists. They showed that once the correct DUST rules were found and applied with a normalization algorithm, they were able to find up to 68% of the redundant URIs in a URI list.
</p>
